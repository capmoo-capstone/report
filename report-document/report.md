# Abstract (Thai)
โครงงานนี้มีชื่อว่า ระบบบริหารจัดการและติดตามงานพัสดุ มุ่งเน้นการแก้ปัญหา ความล่าช้าและปัญหาคอขวดในการติดตามสถานะกระบวนการจัดซื้อจัดจ้าง ซึ่งมีความสำคัญเนื่องจาก กระบวนการจัดซื้อจัดจ้างของมหาวิทยาลัยจำเป็นต้องมีความรวดเร็ว โปร่งใส และสามารถตรวจสอบย้อนหลังได้ วัตถุประสงค์ของโครงงานคือ เพื่อพัฒนาระบบสารสนเทศสำหรับติดตามและบริหารจัดการงานพัสดุ เพื่อตรวจสอบผู้รับผิดชอบและระยะเวลาในการปฏิบัติงานในแต่ละขั้นตอน และเพื่อลดข้อผิดพลาดและเพิ่มประสิทธิภาพในการประสานงานระหว่างส่วนงานต่าง ๆ

แนวทางการแก้ปัญหาประกอบด้วย การนำสถาปัตยกรรมระบบคิวงานกลางและกระบวนการทำงานอัตโนมัติมาประยุกต์ใช้ โดยผสาน สถาปัตยกรรมแบบแยกส่วน การควบคุมสิทธิ์การเข้าถึงตามบทบาท และฐานข้อมูลเชิงสัมพันธ์ ระบบถูกพัฒนาโดยใช้ React, Node.js และ PostgreSQL และผ่านการทดสอบด้วย การประเมินความถูกต้องของการทำงานและการตรวจสอบสิทธิ์การเข้าถึงข้อมูลตามบทบาท

ผลการดำเนินงานแสดงให้เห็นว่า ระบบสามารถเป็นศูนย์กลางในการติดตามสถานะโครงการและกระจายภาระงานได้อย่างเป็นธรรม โดยบรรลุ ความถูกต้องร้อยละ 100 ในกรณีทดสอบระดับวิกฤตที่เกี่ยวข้องกับการตรวจสอบสิทธิ์และการจัดการวงจรสถานะโครงการ โครงงานนี้มีส่วนช่วยต่อ การยกระดับการบริหารจัดการในองค์กร ผ่าน การเปลี่ยนกระบวนการทำงานที่พึ่งพากระดาษไปสู่รูปแบบดิจิทัลที่โปร่งใสและสามารถตรวจสอบย้อนหลังได้

**คำสำคัญ**: ระบบจัดซื้อจัดจ้างอิเล็กทรอนิกส์, ระบบคิวงานกลาง, กระบวนการทำงานอัตโนมัติ, การควบคุมการเข้าถึงตามบทบาท, เว็บแอปพลิเคชัน

# Abstract (English)

This project, titled NexusProcure, addresses the problem of delays and bottlenecks in tracking the status of procurement processes, which is significant due to the necessity for university procurement to be efficient, transparent, and fully traceable. The objectives of this project are to develop an information system for tracking and managing procurement tasks, to monitor responsible personnel and execution time, and to reduce coordination errors.

The proposed solution involves designing a centralized assignment pool and workflow automation, integrating a decoupled architecture, role-based access control, and a relational database. The system was implemented using React, Node.js, and PostgreSQL and evaluated through functional testing and access control security assessments.

Results indicate that the system effectively serves as a centralized hub for tracking project statuses and fairly distributing workloads, achieving a 100% pass rate in critical test cases related to authorization and data integrity. The project contributes to organizational management enhancement by modernizing paper-based workflows into a fully digital, transparent, and verifiable ecosystem.

**Keywords** : e-Procurement, Assignment Pool, Workflow Automation, Role-Based Access Control, Web Application

# 1. บทนำ (3-5 หน้า)

## 1.1 ความเป็นมาและความสำคัญของปัญหา
จุฬาลงกรณ์มหาวิทยาลัย โดยสำนักบริหารการเงิน การบัญชี และการพัสดุ (สบง.) เป็นหน่วยงานหลักที่มีบทบาทสำคัญในการขับเคลื่อนและสนับสนุนการดำเนินงานของมหาวิทยาลัยให้เป็นไปอย่างมีประสิทธิภาพ หนึ่งในภารกิจหลักที่สำคัญคืองานด้านพัสดุ โดยมีฝ่ายการพัสดุเป็นผู้รับผิดชอบหลักในการจัดการ และควบคุมกระบวนการจัดซื้อจัดจ้าง รวมทั้งงานบริหารสัญญา

อย่างไรก็ตาม ปัจจุบันการดำเนินงานของฝ่ายการพัสดุประสบปัญหาที่ส่งผลกระทบต่อประสิทธิภาพและความรวดเร็วในการปฏิบัติงาน ปัญหาหลักประการหนึ่งคือ ความยากลำบากในการติดตามกระบวนการทำงานของเจ้าหน้าที่พัสดุในแต่ละขั้นตอน เนื่องจากยังไม่มีระบบเทคโนโลยีสารสนเทศที่เข้ามาช่วยบริหารจัดการ และติดตามสถานะของโครงการอย่างเป็นรูปธรรม การติดตามความคืบหน้าของงานส่วนใหญ่ยังคงใช้ระบบเอกสารแบบกระดาษ ส่งผลให้เกิดข้อจำกัดในกรณีที่ต้องการข้อมูลแบบ real-time ทำให้ผู้บริหารและหัวหน้ากลุ่มงานไม่สามารถรับรู้ถึงรายละเอียดต่าง ๆ ของแต่ละโครงการได้อย่างทันที ซึ่งอาจส่งผลให้เกิดปัญหาปริมาณงานสะสมและงานค้างเป็นจำนวนมาก อีกทั้งการตรวจสอบปริมาณงานที่เสร็จสิ้นแล้วกระทำได้ยากและล่าช้า นอกจากนี้ปัญหาดังกล่าวยังทวีความรุนแรงขึ้น เมื่อเกิดความคลาดเคลื่อนในการประสานงานระหว่างหน่วยงานภายในมหาวิทยาลัย ซึ่งส่งผลให้ฝ่ายการพัสดุไม่สามารถดำเนินการต่อไปได้ ทำให้กระบวนการทั้งหมดต้องหยุดชะงักและเกิดความล่าช้าในภาพรวม

จากปัญหาที่กล่าวมาข้างต้น ฝ่ายการพัสดุ สบง. ร่วมกับสาขาวิชาวิศวกรรมคอมพิวเตอร์และเทคโนโลยีดิจิทัล คณะวิศวกรรมศาสตร์ จุฬาลงกรณ์มหาวิทยาลัย เล็งเห็นถึงความจำเป็นเร่งด่วนในการพัฒนาระบบเทคโนโลยีสารสนเทศเพื่อเข้ามาแก้ไขปัญหาดังกล่าว จึงเกิดเป็นแนวคิดในการพัฒนาเว็บแอปพลิเคชัน NexusProcure ขึ้น โดยมุ่งเน้นเป็นระบบส่วนกลางในการบริหารจัดการกระบวนการจัดซื้อจัดจ้างและบริหารสัญญาแบบครบวงจร ช่วยยกระดับการบริหารจัดการงานพัสดุให้มีความทันสมัย โปร่งใส ตรวจสอบได้ และเพิ่มประสิทธิภาพในการให้บริการแก่ทุกภาคส่วนภายในมหาวิทยาลัยได้อย่างยั่งยืน

## 1.2 คำถามโครงงาน
ปัญหาหลักที่โครงการนี้มุ่งแก้ไขคือ ความล่าช้าและความยากลำบากในการติดตามสถานะของกระบวนการจัดซื้อจัดจ้างและการบริหารสัญญา ซึ่งปัจจุบันยังคงพึ่งพาระบบการทำงานแบบกระดาษ ทำให้ผู้บริหาร หัวหน้ากลุ่มงาน และผู้ปฏิบัติงาน ไม่สามารถทราบข้อมูลต่าง ๆ แบบ real-time คณะผู้จัดทำพบว่าปัญหาดังกล่าวมีความซับซ้อนและแก้ไขได้ยากด้วยวิธีการแบบเดิม เนื่องจากปัจจัยต่อไปนี้
1. ความคลาดเคลื่อนในการประสานงานระหว่างหลายภาคส่วน ซึ่งบ่อยครั้งเกิดจากความผิดพลาดในการสื่อสาร ทำให้การทำงานของเจ้าหน้าที่ต้องหยุดชะงัก เกิดความล่าช้าในภาพรวม
2.	การเกิดปัญหาคอขวดจากข้อจำกัดจากการทำงานแบบกระดาษ ส่งผลให้เกิดปริมาณงานค้างสะสม
3.	ความเข้มงวดของกฎระเบียบภาครัฐ เนื่องจากงานจัดซื้อจัดจ้างจำเป็นต้องดำเนินการภายใต้กฎระเบียบต่าง ๆ จึงต้องออกแบบระบบมาให้มีความรัดกุม โปร่งใส และสามารถตรวจสอบย้อนหลังได้

## 1.3 วัตถุประสงค์โครงงาน
1.	เพื่อพัฒนาระบบสารสนเทศสำหรับติดตามและบริหารจัดการงานด้านพัสดุ ผ่านการจัดทำเว็บแอปพลิเคชันที่สามารถติดตามสถานะของกระบวนการจัดซื้อจัดจ้างในแต่ละขั้นตอน เพื่อเป็นการทดแทนการติดตามงานด้วยกระดาษแบบเดิม
2.	เพื่อตรวจสอบผู้รับผิดชอบและระยะเวลาในการปฏิบัติงาน ทำให้สามารถระบุตัวเจ้าหน้าที่ผู้รับผิดชอบงานในแต่ละขั้นตอน และบันทึกระยะเวลาที่ใช้ในการดำเนินการ ลดปัญหาคอขวดและปริมาณงานค้างสะสม
3.	เพื่อลดข้อผิดพลาดและเพิ่มประสิทธิภาพในการประสานงานระหว่างส่วนงานต่าง ๆ ในมหาวิทยาลัย ให้มีระบบที่ช่วยตรวจสอบความถูกต้องของเอกสาร ซึ่งจะช่วยลดความล่าช้าในการดำเนินงาน

## 1.4 ขอบเขตและข้อจำกัดของโครงงาน
โครงงานเรื่อง NexusProcure (ระบบฯ) มีการจำกัดขอบเขตของการพัฒนาเว็บแอปพลิเคชันเอาไว้ 4 ข้อหลัก รวมถึงข้อจำกัดที่ไม่ใช่ฟังก์ชันการทำงาน ดังนี้
1. ระบบยืนยันตัวตนและจัดการผู้ใช้งาน
    1. ระบบฯ ต้องรองรับการยืนยันตัวตนผู้ใช้งานภายใน เพื่ออนุญาตให้ผู้ปฏิบัติงานและเจ้าหน้าที่แต่ละฝ่ายเข้าใช้งานระบบได้อย่างปลอดภัย
    2. ระบบฯ กำหนดให้ 1 ฝ่าย มีผู้ใช้งานในบทบาทผู้แทนฝ่าย ได้เพียง 1 คนเท่านั้น เพื่อป้องกันความซ้ำซ้อนในการสร้างโครงการ
    3. ระบบฯ จะมีหน้าจอสำหรับผู้ดูแลระบบเพื่อเชื่อมโยงบัญชีผู้ใช้เข้ากับฝ่ายและกำหนดบทบาทการทำงาน
    4. ระบบฯ ต้องสามารถระบุผู้รักษาการแทนชั่วคราว เพื่อให้มีสิทธิ์ในการดำเนินงานแทนเจ้าหน้าที่ตัวจริงในช่วงเวลาที่กำหนด หรือกรณีที่มีการเปลี่ยนแปลงหัวหน้างาน
    5. ระบบฯ ต้องอนุญาตให้ผู้ดูแลระบบสามารถสร้าง แก้ไข หรือลบ กลุ่มงานภายในหน่วยงานพัสดุได้
2. ระบบการรับเรื่องและมอบหมายงาน
    1. ผู้แทนหน่วยงานหรือเจ้าหน้าที่งานเอกสาร สามารถสร้างโครงการได้ 2 รูปแบบ คือ สร้างใหม่จากฟอร์มภายในระบบฯ หรือเลือกจากแผนรายงานจัดซื้อจัดจ้าง ครุภัณฑ์ และสิ่งก่อสร้างประจำปีงบประมาณ (แผนฯ) ที่นำเข้าไว้แล้วในรูปแบบการเลือกหลายรายการ (Multi-select) โดยระบบฯ จะคำนวณระดับความเร่งด่วนของงานให้อัตโนมัติโดยอ้างอิงจากวันทำการและวันหยุดนักขัตฤกษ์ หากเวลาดำเนินการน้อยกว่าเกณฑ์ ระบบจะแสดงข้อความแจ้งเตือนให้ยืนยัน รวมทั้งระบบฯ จะออกเลขที่ลงรับอัตโนมัติ และเก็บเลขอ้างอิงจากระบบเดิมไว้ โดยกำหนดให้ 1 โครงการ มี 1 เลขที่ลงรับเท่านั้น
    2. ระบบฯ ต้องรองรับการนำเข้าข้อมูลหลายโครงการพร้อมกันผ่านไฟล์รูปแบบ Microsoft Excel โดยระบบจะตรวจสอบความถูกต้องของข้อมูล เช่น การซ้ำซ้อนของเลขที่อ้างอิง วันที่ส่งมอบงาน หรือวงเงิน และอนุญาตให้แก้ไขหรือลบข้อมูลก่อนยืนยันการบันทึก
    3. เมื่อมีการสร้างโครงการ ระบบฯ จะส่งงานเข้าสู่ระบบงานกลาง (Assignment Pool) ของกลุ่มงานที่เกี่ยวข้องโดยอัตโนมัติ ในสถานะงาน ยังไม่ได้มอบหมาย
    4. เจ้าหน้าที่ในกลุ่มงานสามารถเลือกงานที่ตนเองถนัดหรือตามความเหมาะสมจากระบบงานกลางมาเป็นงานในความรับผิดชอบได้
    5. หัวหน้ากลุ่มงานสามารถมอบหมายงานโดยระบุตัวบุคคลได้ แต่เจ้าหน้าที่ต้องกดยืนยันการรับงานดังกล่าวเพื่อเริ่มการทำงานจริงและเปลี่ยนสถานะเป็นกำลังดำเนินการ
    6. ระบบฯ ต้องแสดงผลข้อมูลในรูปแบบภาพรวมเพื่อให้หัวหน้ากลุ่มงานสามารถตรวจสอบปริมาณงานแบ่งตามสถานะที่เจ้าหน้าที่แต่ละคนกำลังรับผิดชอบอยู่ เพื่อใช้เป็นข้อมูลประกอบการตัดสินใจมอบหมายงาน
    7. ระบบฯ ต้องแสดงรายการงานส่วนบุคคลที่ผู้ใช้งานต้องดำเนินการ ได้แก่ งานที่ได้รับมอบหมายใหม่ งานที่ถูกส่งกลับให้แก้ไข และงานเร่งด่วนที่ใกล้ถึงกำหนดชำระหรือครบกำหนดสัญญา
    8. ระบบฯ ต้องรองรับการนำเข้าข้อมูลแผนฯ ของแต่ละหน่วยงานผ่านไฟล์รูปแบบ Microsoft Excel เพื่อใช้เป็นฐานข้อมูลตั้งต้น โดยรายการงบประมาณที่ถูกเชื่อมโยงกับโครงการอื่นไปแล้ว จะต้องไม่แสดงซ้ำในรายการเลือกสำหรับสร้างโครงการใหม่
    9. ระบบฯ จะต้องสามารถเปลี่ยนผู้รับผิดชอบของโครงการได้ในกรณีที่งานอยู่ในสถานะรอการตอบกลับ และทำการส่งการแจ้งเตือนให้เจ้าหน้าที่คนเก่ารับทราบ
    10. ระบบฯ จะต้องสามารถเพิ่มผู้รับผิดชอบของโครงการได้ในขณะที่สถานะกำลังดำเนินการ โดยกำหนดให้ 1 โครงการมีผู้รับผิดชอบได้มากสุดไม่เกิน 2 คน ต่อ 1 ช่วงการทำงาน
3. ระบบกระบวนการทำงานจัดซื้อจัดจ้าง
    1. ระบบฯ ต้องควบคุมขั้นตอนงานและหน้าจอขั้นตอนย่อยให้สอดคล้องกับวิธีจัดซื้อและบริหารสัญญา
    2. เจ้าหน้าที่สามารถบันทึกความคืบหน้าของงาน และหัวหน้ากลุ่มงานสามารถสั่งส่งงานกลับพร้อมระบุเหตุผล เพื่อให้แก้ไขข้อมูลต่าง ๆ ได้ พร้อมบันทึกประวัติการส่งคืนงาน
    3. ระบบฯ ต้องรองรับการจัดเก็บเอกสารประกอบโครงการแยกตามขั้นตอน โดยสามารถอัปโหลดไฟล์ได้หลายประเภท และต้องมีระบบจัดเก็บประวัติไฟล์ เพื่อให้สามารถตรวจสอบ สร้างไฟล์รายงานลำดับเหตุการณ์ และส่งออกชุดไฟล์เอกสารรวมให้กับฝ่ายอื่น ๆ ที่เกี่ยวข้อง
    4. ระบบฯ ต้องมีหน้าจอสำหรับเจ้าหน้าที่หน่วยงานอื่น เพื่อใช้ติดตามความคืบหน้าของโครงการที่หน่วยงานตนเองมีความเกี่ยวข้อง
    5. ระบบฯ ต้องสามารถรวบรวมไฟล์แนบและข้อมูลสรุปจากทุกขั้นตอนมาแสดงผลในหน้าเดียว เพื่อให้ผู้บริหารตรวจสอบและอนุมัติได้สะดวก
    6. ระบบฯ ต้องรองรับการสร้างเลขที่สัญญาตามรูปแบบ เลขที่/ปีงบประมาณ โดยเริ่มนับใหม่ทุกวันที่ 1 ตุลาคม เจ้าหน้าที่สามารถเลือกปีงบประมาณในอนาคตได้ และเลขที่สัญญามีทั้งหมด 5 ชุด แบ่งตามแผนกและสำนักงานมหาวิทยาลัย แต่การแสดงผลจะไม่ระบุชื่อแผนกนำหน้า โดยต้องสามารถยกเลิกเลขที่สัญญานี้ได้ พร้อมบันทึกเหตุผล
    7. เมื่อสิ้นสุดขั้นตอนการจัดซื้อ และก่อนที่จะเปลี่ยนสถานะเข้าสู่ขั้นตอนการบริหารสัญญา ระบบฯ จะต้องแสดงหน้าต่างเพื่อสอบถามผู้ใช้งานว่าโครงการนี้มีรหัสสินทรัพย์หรือไม่
    8. เจ้าหน้าที่พัสดุสามารถยกเลิกโครงการใดก็ได้ โดยจะต้องระบุเหตุผล และส่งให้หัวหน้ากลุ่มงานอนุมัติ
4. ระบบวางบิลสำหรับผู้ขาย
    1. ผู้ขายสามารถอัปโหลดไฟล์หลักฐานการส่งมอบงานผ่านหน้าเว็บได้โดยตรวจสอบผ่านเลขใบสั่งซื้อ หรือ Purchase Order (PO)
5. ขอบเขตความต้องการอื่น ๆ
    1. การออกแบบส่วนติดต่อผู้ใช้งานต้องเน้นตัวอักษรที่มีขนาดใหญ่ สีเข้ม และมีความหนา เพื่อให้อ่านง่ายและชัดเจน
    2. ระบบต้องบันทึกเวลาที่เกิดขึ้นจริงของระบบ (System Time) เมื่อมีการแก้ไขข้อมูลเพื่อเก็บเป็นประวัติการแก้ไขงาน แต่ในส่วนของการทำงานตามกระบวนการ ระบบต้องอนุญาตให้ผู้ใช้ระบุวันที่ย้อนหลังได้ เพื่อความสอดคล้องกับเอกสารฉบับจริง

## 1.5 ผู้มีส่วนได้ส่วนเสียกับโครงงาน
1. กลุ่มผู้ใช้งานระบบโดยตรง หรือ end-users
    1. เจ้าหน้าที่พัสดุและหัวหน้ากลุ่มงาน เป็นกลุ่มผู้ใช้งานหลักที่จะได้ประโยชน์จากการเปลี่ยนผ่านจากการทำงานในระบบกระดาษสู่ระบบดิจิทัลโดยระบบงานกลาง ที่เข้ามาช่วยขจัดปัญหาความเหลื่อมล้ำของการกระจายงาน ทำให้หัวหน้างานสามารถประเมินและบริหารจัดการทรัพยากรบุคคลได้อย่างมีประสิทธิภาพสูงสุด
    2. ผู้บริหารสำนักพัสดุ ได้รับเครื่องมือสนับสนุนการตัดสินใจในภาพรวม ทำให้สามารถมองเห็นปัญหาคอขวดที่เกิดขึ้น และสามารถนำข้อมูลจากเว็บไซต์ไปใช้ในการวางแผนต่อไป
2. กลุ่มผู้ใช้งานระดับหน่วยงานภายในมหาวิทยาลัย
    1. ผู้แทนหน่วยงานผู้ขอซื้อ/จ้าง  โดยระบบจะช่วยลดภาระและความซ้ำซ้อนในการติดตามงาน โดยผู้แทนหน่วยงานสามารถเข้าสู่ระบบเพื่อตรวจสอบสถานะโครงการของตนเองได้แบบ real-time ช่วยลดความคลาดเคลื่อนในการสื่อสารระหว่างหน่วยงานและเพิ่มความพึงพอใจในการรับบริการจากฝ่ายการพัสดุ
3. กลุ่มผู้มีส่วนได้เสียภายนอก
    1. ผู้ขายและบริษัทคู่ค้า จะได้รับประสบการณ์การประสานงานที่รวดเร็วและเป็นระบบมากขึ้นผ่านระบบบริการสำหรับผู้ขายที่รองรับการอัปโหลดเอกสารส่งมอบงานทางออนไลน์


## 1.6 ประโยชน์ที่คาดว่าจะได้รับ
1. ช่วยยกระดับการติดตามสถานะโครงการอย่างเป็นรูปธรรม โดยผู้ใช้งานและผู้บริหารสามารถติดตามสถานะของกระบวนการจัดซื้อจัดจ้างในแต่ละขั้นตอนได้แบบ real-time ซึ่งเข้ามาทดแทนการใช้ระบบเอกสารแบบเดิม
2. ช่วยลดปัญหาคอขวดและปริมาณงานค้างสะสม โดยระบบนี้จะเข้ามาช่วยระบุตัวเจ้าหน้าที่ผู้รับผิดชอบงานและบันทึกระยะเวลาที่ใช้ในการดำเนินการแต่ละขั้นตอนได้อย่างชัดเจน ทำให้สามารถติดตามและจัดการปัญหาความล่าช้าหรือปริมาณงานที่สะสมอยู่ได้อย่างทันท่วงที
3. ช่วยเพิ่มประสิทธิภาพการประสานงานและลดข้อผิดพลาด โดยฝ่ายการพัสดุจะมีระบบที่ช่วยตรวจสอบความถูกต้องของเอกสาร ซึ่งช่วยลดปัญหาความคลาดเคลื่อนและการส่งเอกสารไม่ครบถ้วนระหว่างหน่วยงาน ทำให้กระบวนการดำเนินงานลื่นไหลและไม่หยุดชะงัก
4. ช่วยสร้างมาตรฐานใหม่ในการบริหารจัดการงานพัสดุ ของฝ่ายการพัสดุ ให้มีความทันสมัย โปร่งใส และตรวจสอบได้ โดยทำหน้าที่เป็นเครื่องมือส่วนกลางที่ครอบคลุมการจัดการแบบครบวงจรและเพิ่มประสิทธิภาพการให้บริการอย่างยั่งยืน

# 2. ทฤษฎีและงานวิจัยที่เกี่ยวข้อง (3-5 หน้า)

## 2.1 ทบทวนองค์ความรู้ที่มีอยู่
ระบบจัดซื้อจัดจ้างแบบอิเล็กทรอนิกส์ หรือ e-Procurement เป็นระบบที่เข้ามาแทนที่ระบบจัดซื้อจัดจ้างแบบเดิม มักถูกใช้ในองค์กรต่าง ๆ รวมถึงสถาบันการศึกษาด้วย ระบบนี้ถูกสร้างขึ้นมาเพื่อเพิ่มประสิทธิภาพ ความโปร่งใส และความสามารถในการตรวจสอบย้อนหลัง [1, 2] ภายในระบบมักจะแบ่งกระบวนการจัดซื้อจัดจ้างออกเป็นขั้นตอนต่าง ๆ ได้แก่ การตรวจสอบคุณสมบัติเบื้องต้น การประกาศเชิญชวนทั่วไป การยื่นข้อเสนอแนะหรือใบเสนอราคา การประเมินผล และการประกาศผลผู้ชนะการจัดซื้อจัดจ้าง [3]

เพื่อให้การดำเนินงานในแต่ละขั้นตอนมีความลื่นไหลและลดปัญหาความล่าช้า องค์กรสมัยใหม่ได้นำแนวคิดเรื่อง Automation Workflow และ Business Process Management (BPM) เข้ามาประยุกต์ใช้ งานวิจัยชี้ให้เห็นว่าการเปลี่ยนผ่านจากการดำเนินงานแบบพึ่งพากระดาษเป็นหลัก มาเป็นการใช้ระบบงานกลาง และการติดตามสถานะแบบออนไลน์ สามารถลดระยะเวลารอคอย และลดปัญหาคอขวดในกระบวนการดำเนินงานได้อย่างมีนัยสำคัญ ระบบที่ดีจะช่วยให้ผู้ปฏิบัติงานทราบถึงลำดับความสำคัญของงานและระยะเวลาที่ใช้ในแต่ละขั้นตอนอย่างชัดเจน [4]

นอกจากนี้ ในระบบสารสนเทศระดับองค์กรที่มีผู้ใช้งานหลากหลายและมีโครงสร้างที่ซับซ้อน การจัดการสิทธิ์การเข้าถึงข้อมูลจึงมีความสำคัญอย่างยิ่ง จึงมีการนำวิธีการของ Role-Based Access Control (RBAC) เข้ามาใช้เป็นมาตรฐานในการออกแบบระบบรักษาความปลอดภัย โดย RBAC จะสร้างความเชื่อมโยงระหว่างผู้ใช้งาน บทบาทหน้าที่ และสิทธิ์การอนุญาตเข้าด้วยกัน ทำให้การดูแลระบบมีความยืดหยุ่นสูง หากมีการโยกย้ายตำแหน่งงานภายในองค์กร ผู้ดูแลระบบสามารถปรับเปลี่ยนบทบาทของผู้ใช้งานได้ทันทีโดยไม่ต้องแก้ไขสิทธิ์ในแต่ละฟังก์ชัน ซึ่งในบริบทของระบบจัดซื้อจัดจ้างที่มีความเกี่ยวข้องกับบุคลากรหลายระดับ ได้แก่ เจ้าหน้าที่พัสดุ หัวหน้ากลุ่มงาน ผู้บริหาร ผู้แทนหน่วยงานผู้ขอ และผู้ขาย การประยุกต์ใช้แนวคิด RBAC ถือเป็นการช่วยลดความซ้ำซ้อนในการบริหารจัดการสิทธิ์ และช่วยเพิ่มความปลอดภัยสูงสุดให้กับข้อมูลที่มีความละเอียดอ่อนในกระบวนการประมูลหรือจัดซื้อ [5] นอกจากนี้ยังมีงานวิจัยที่สนับสนุนว่า การนำเทคโนโลยีจัดการสิทธิ์สมัยใหม่ เช่น JSON Web Token (JWT) มาประยุกต์ใช้ร่วมกับกลไก RBAC จะยิ่งช่วยให้การยืนยันตัวตนและการตรวจสอบสิทธิ์ผ่าน Application Programming Interface (API) มีความรัดกุม รวดเร็ว และสามารถตรวจสอบประวัติการเข้าถึงได้อย่างแม่นยำ [6]

ในบริบทของประเทศไทย มีการยกระดับและควบคุมการจัดซื้อจัดจ้างภาครัฐผ่านพระราชบัญญัติการจัดซื้อจัดจ้างและการบริหารพัสดุภาครัฐ พ.ศ. 2560 โดยมีกรมบัญชีกลางเป็นหน่วยงานหลักในการพัฒนาระบบ e-GP หรือ Electronic Government Procurement ให้เป็นศูนย์กลางข้อมูลสารสนเทศ เพื่อสร้างความโปร่งใส ป้องกันการทุจริตคอร์รัปชัน และเปิดโอกาสให้ผู้มีส่วนได้ส่วนเสียสามารถตรวจสอบข้อมูลได้ [7] อย่างไรก็ตาม ถึงแม้ระบบ e-GP จะเป็นเครื่องมือหลักในการจัดซื้อจัดจ้างระดับประเทศ แต่ในระดับการปฏิบัติงานภายในองค์กรขนาดใหญ่ เช่น มหาวิทยาลัยหรือหน่วยงานราชการมักจะมีกระบวนการอนุมัติและเตรียมเอกสารภายในที่ซับซ้อน และต้องผ่านสายการบังคับบัญชาหลายขั้นตอนก่อนที่จะสามารถนำข้อมูลเข้าสู่ระบบ e-GP ได้ ความเข้มงวดของกฎระเบียบนี้ส่งผลให้เกิดการพึ่งพาเอกสารกระดาษในขั้นตอนเตรียมการ ซึ่งเป็นสาเหตุหลักของความล่าช้าอย่างมากในระดับองค์กร และยังพบว่ากระบวนการทำงานภายในมหาวิทยาลัยมักเกิดความล่าช้าสะสม โดยที่ผู้บริหารไม่สามารถทราบสาเหตุที่แน่ชัดว่าเกิดจากขั้นตอนใดหรืออยู่ในการรับผิดชอบของเจ้าหน้าที่ท่านใด [8]

## 2.2 เปรียบเทียบกับระบบที่มีอยู่
จากการศึกษาระบบ e-Procurement ที่มีอยู่ในปัจจุบัน จากงานวิจัยต่าง ๆ ที่ศึกษาเรื่องดังกล่าว คณะผู้จัดทำพบข้อดีและข้อเสียของระบบดังกล่าว โดยจะขอกล่าวในจุดเด่นก่อน คือ ระบบ e-Procurement ช่วยเพิ่มความโปร่งใสและลดโอกาสในการทุจริตได้อย่างเป็นรูปธรรม เนื่องจากระบบมีคุณสมบัติในการตรวจสอบย้อนหลัง ทำให้พบว่าสามารถลดโอกาสที่จะเกิดการประพฤติมิชอบลงได้อย่างเห็นได้ชัด [9] นอกจากนี้ การนำระบบอิเล็กทรอนิกส์มาใช้ยังช่วยเสริมประสิทธิภาพในการดำเนินงานโดยลดระยะเวลาในการรับส่งข้อมูลและงานด้านเอกสาร ส่งผลให้ต้นทุนการดำเนินงานโดยรวมลดลงอย่างมีนัยสำคัญ [10, 11]

ในด้านเทคโนโลยีการพัฒนาเว็บแอปพลิเคชัน การนำเครื่องมือและเฟรมเวิร์กสมัยใหม่ที่เรียกกันว่า component-based architecture มาประยุกต์ใช้ ถือเป็นการแก้ไขข้อจำกัดด้านประสบการณ์ผู้ใช้งานและการบำรุงรักษาระบบ e-Procurement ในอดีตได้อย่างก้าวกระโดด รูปแบบการพัฒนานี้ช่วยให้ระบบมีโครงสร้างที่ชัดเจน สามารถนำส่วนประกอบกลับมาใช้ซ้ำ และจัดการสถานะของหน้าจอได้ดีขึ้น โดยการเลือกใช้เทคโนโลยี React สำหรับการสร้างส่วนติดต่อผู้ใช้งาน และใช้ Node.js สำหรับส่วนประมวลผล ทำให้ระบบสามารถรองรับการจัดการข้อมูลที่ซับซ้อน และการเปลี่ยนเส้นทางหน้าเว็บไซต์ได้อย่างมีประสิทธิภาพและมีเสถียรภาพสูง [13]

อย่างไรก็ตาม ระบบที่มีอยู่ในปัจจุบันยังคงมีจุดด้อยหลายประการ ประการแรกคือความครอบคลุมของกระบวนการที่จำกัด หลายระบบรองรับเพียงการทำงานบางส่วน เช่น การจัดทำแผนงบประมาณหรือการวางแผนจัดซื้อ โดยมักขาดระบบปฏิบัติงานอัตโนมัติที่ครอบคลุมไปถึงการบริหารสัญญาและการวางบิล [14] ประการที่สองคือข้อจำกัดเรื่องโครงสร้างพื้นฐานทางเทคโนโลยีและทักษะของบุคลากร โดยปัญหาที่พบบ่อยคือ server แบบเดิมไม่สามารถรองรับภาระงานหนักได้ ทำให้การอัปโหลดหรือดาวน์โหลดไฟล์เอกสารแนบขนาดใหญ่เกิดความล่าช้าจนผู้ใช้งานเกิดความหงุดหงิด รวมถึงเจ้าหน้าที่หรือผู้ใช้งานบางส่วนยังขาดความคุ้นเคยกับเทคโนโลยี [9]

นอกจากนี้ งานวิจัยล่าสุดยังชี้ให้เห็นถึงปัญหาด้านความซับซ้อนและด้าน User Experience (UX) ของการใช้งานระบบดังกล่าว โดยพบว่าความไม่พึงพอใจในระบบ e-Procurement ภาครัฐมักมีสาเหตุหลักมาจากการขาดการออกแบบ User Interface (UI) ที่เป็นมิตรและใช้งานง่าย ซึ่งส่งผลให้เกิดแรงต้านในการยอมรับการเปลี่ยนแปลงเทคโนโลยีของบุคลากร [15] และประการสุดท้ายคือ การขาดความเชื่อมโยงในกระบวนการทำงานแบบองค์รวมที่ส่งผลต่อปัญหาคอขวด หากการออกแบบระบบอิเล็กทรอนิกส์ไม่สอดคล้องกับขั้นตอนการทำงานจริงมักจะทำให้เกิดช่องว่างระหว่างขั้นตอน เช่น ช่วงรอยต่อระหว่างการแจ้งผลผู้ชนะการเสนอราคาและการเริ่มบริหารสัญญา ส่งผลให้ปัญหาภาระงานของเจ้าหน้าที่พัสดุไม่สามารถแก้ไขและเพิ่มประสิทธิภาพได้อย่างยั่งยืน [16]

## 2.3 การเชื่อมโยงช่องโหว่กับโครงงาน
จากการศึกษาทฤษฎีและงานวิจัยที่เกี่ยวข้อง พบว่าถึงแม้จะมีการประยุกต์ใช้งานระบบ e-Procurement อย่างแพร่หลาย แต่ก็ยังมีช่องโหว่อีกหลายจุดที่ระบบทั่วไปยังไม่สามารถแก้ไขได้สำเร็จ คณะผู้จัดทำจึงได้ออกแบบและพัฒนาโครงงาน NexusProcure โดยมุ่งเน้นการใช้ software architecture สมัยใหม่เพื่อเข้ามาปิดช่องโหว่เหล่านี้ ได้แก่

1.	ช่องโหว่ในการตรวจสอบเอกสารการตัดสินใจ เราพบว่าระบบเดิมมักประสบปัญหาการจัดเก็บและนำเสนอข้อมูลเอกสารที่กระจัดกระจาย ทำให้หัวหน้างานและผู้บริหารต้องใช้เวลามากในการค้นหาเอกสารเพื่อประกอบการตัดสินใจ โครงงานนี้แก้ไขปัญหาดังกล่าวโดยพัฒนาระบบรวบรวมไฟล์แนบและข้อมูลสรุปจากทุกขั้นตอนกระบวนการจัดซื้อมาแสดงผลภายในหน้าเดียว ทำให้การตรวจสอบความถูกต้องของเอกสารอ้างอิงมีความสะดวกรวดเร็ว ลดความผิดพลาด และสนับสนุนการอนุมัติงานได้อย่างมีประสิทธิภาพ
2.	ช่องโหว่ในการบริหารจัดการวงจรจัดซื้อแบบครบวงจรและเชื่อมโยงผู้มีส่วนได้ส่วนเสีย เราพบว่าระบบ e-Procurement ทั่วไปยังมีข้อจำกัดเรื่องความครอบคลุมของกระบวนการ โครงงานนี้จึงออกแบบระบบให้เป็นศูนย์กลางที่เชื่อมโยงทุกภาคส่วนเข้าด้วยกัน ตั้งแต่การให้ผู้แทนหน่วยงานนำเข้าข้อมูลแผนฯ การออกเลขที่ลงรับโครงการ การจัดการกระบวนการจัดซื้อ ไปจนถึงการสร้างเลขที่สัญญา และครอบคลุมไปถึงการสร้าง portal สำหรับผู้ขาย ที่อนุญาตให้บริษัทคู่ค้าสามารถตรวจสอบสถานะ และอัปโหลดหลักฐานการส่งมอบงานหรือเอกสารวางบิลเข้าสู่ระบบได้โดยตรง ทำให้เจ้าหน้าที่พัสดุสามารถติดตามเอกสาร และจัดการกระบวนการต่าง ๆ ได้อย่างครบถ้วนและมีประสิทธิภาพมากขึ้น
3.	ช่องโหว่ในการติดตามสถานะการดำเนินงาน โครงงานนี้ได้นำหลักการ BPM มาประยุกต์ใช้ผ่านการสร้างระบบงานกลาง เพื่อแก้ไขปัญหาความล่าช้าในขั้นตอนการทำงานโดยไม่ทราบสาเหตุและไม่ทราบผู้รับผิดชอบ รวมทั้งการจัดการ และกระจายงานให้กับเจ้าหน้าที่พัสดุอย่างเป็นธรรม พร้อมทั้งการพัฒนาระบบจัดเก็บประวัติสถานะโครงการที่สามารถบันทึก history logs แบบอัตโนมัติ ทำให้ทั้งผู้ปฏิบัติงาน หน่วยงานผู้ขอ และผู้บริหารสามารถติดตามความคืบหน้าของโครงการได้อย่างโปร่งใส และสามารถระบุขั้นตอนที่เป็นปัญหาคอขวดได้อย่างแม่นยำ

# 3. การออกแบบระบบ (5-8 หน้า)

## 3.1 สถาปัตยกรรมระบบ
คณะผู้จัดทำได้ออกแบบระบบ NexusProcure ให้ใช้สถาปัตยกรรมแบบ Client-Server Model เพื่อทำให้การพัฒนา การทดสอบ และการดูแลรักษาระบบเป็นไปอย่างมีประสิทธิภาพและสามารถขยายขนาดของระบบได้ในอนาคต โดยสถาปัตยกรรมหลักประกอบด้วย 3 ส่วนสำคัญ ได้แก่

1. Presentation Layer หรือ frontend ทำหน้าที่แสดงผลและรับคำสั่งจากผู้ใช้งาน โดยแบ่งการจัดการ state ออกเป็น 2 ส่วน คือ server state และ client state เพื่อลดภาระการประมวลผลและลดการดึงข้อมูลที่ซ้ำซ้อน
2. Application Layer หรือ backend ทำหน้าที่ประมวลผล business logic ตรวจสอบสิทธิ์การเข้าถึงแบบ RBAC และเป็นตัวกลางในการสื่อสารกับฐานข้อมูลผ่าน RESTful API
3. Data Layer หรือ database ทำหน้าที่จัดเก็บข้อมูลทั้งหมดของระบบในรูปแบบของ relational database

โดยมีลักษณะของ data flow ดังนี้
1. ผู้ใช้งาน (เจ้าหน้าที่/หัวหน้ากลุ่มงาน/ผู้บริหาร/ผู้แทนฝ่าย/ผู้ขาย) เข้าใช้งานผ่านเว็บแอปพลิเคชัน
2. Frontend ส่งคำขอผ่าน REST API ไปยัง backend
3. Backend ตรวจสอบสิทธิ์ด้วย JWT และบทบาทผู้ใช้งาน RBAC
4. Service Layer ประมวลผลตาม workflow ของโครงการจัดซื้อจัดจ้าง
5. บันทึกหรืออ่านข้อมูลจาก PostgreSQL ผ่าน Prisma ORM
6. ส่งผลลัพธ์กลับไปแสดงผลเป็นตารางและ Timeline ที่ frontend

เพื่อรองรับงานจัดซื้อจัดจ้างที่มีหลายขั้นตอน ระบบฯ กำหนดสถานะโครงการแบบแยกช่วงการดำเนินงานเป็นช่วงจัดซื้อและช่วงบริหารสัญญา พร้อมลำดับขั้นตอนงานตามประเภทการจัดซื้อ และมีการบันทึก log เพื่อการตรวจสอบย้อนหลัง

## 3.2 องค์ประกอบ
องค์ประกอบหลักของระบบแบ่งออกเป็น 3 ส่วนหลัก ดังแสดงในภาพที่ 3.2 ประกอบด้วย

1. ส่วน frontend พัฒนาด้วย React (Vite) และ TypeScript โดยมีการจัดโครงสร้างแบบ Feature-Based Architecture เพื่อแยกองค์ประกอบตามโดเมนงานออกจากกันอย่างชัดเจน มีการใช้ Tailwind CSS ร่วมกับ shadcn/ui ในการออกแบบและสร้างส่วน UI ที่สวยงามและเป็นมาตรฐาน นอกจากนี้ยังใช้ TanStack Query ในการจัดการข้อมูลและ cache ฝั่ง client และใช้ TanStack Table สำหรับการจัดการแสดงผลตารางข้อมูลที่ซับซ้อน เช่น ตารางโครงการ หรือตารางรายชื่อเจ้าหน้าที่
2. ส่วน backend พัฒนาด้วย Node.js ร่วมกับ Express มีการควบคุมความถูกต้องของข้อมูล ทั้งฝั่งรับและส่งผ่าน Zod และจัดการการยืนยันตัวตนผ่าน JWT ร่วมกับระบบ in-memory cache แบบ Least Recently Used (LRU)  สำหรับจัดเก็บข้อมูลที่มีการเข้าถึงบ่อยเพื่อลดภาระของฐานข้อมูล
3. ระบบฐานข้อมูล ใช้ PostgreSQL เป็นฐานข้อมูลหลัก โดยมี Prisma ORM เป็นเครื่องมือในการจัดการ schema และ query ข้อมูลต่าง ๆ อย่างมีประสิทธิภาพและมีความปลอดภัยแบบ type-safety โดยมีการออกแบบระบบฐานข้อมูล ตามแผนภาพ Entity Relationship (ER) ดังแสดงในภาพที่ 3.4

## 3.3 การพิจารณาการออกแบบ
เพื่อให้ระบบสามารถรองรับการใช้งานจริงของฝ่ายการพัสดุ ได้อย่างมีประสิทธิภาพ คณะผู้จัดทำได้คำนึงถึงปัจจัยสำคัญในการออกแบบ ดังนี้

1. ด้านประสิทธิภาพ โดยการลดภาระการประมวลผลที่ไม่จำเป็นด้วยการแบ่งหน้าข้อมูลหรือ pagination มีการกำหนด filter และ query ที่ชัดเจน รวมทั้งการ cache ข้อมูลฝั่ง backend บางส่วน เช่น ข้อมูลสิทธิ์ผู้ใช้ เพื่อให้รองรับข้อมูลโครงการจำนวนมาก
2. ด้านการขยายขนาดของเว็บไซต์ผ่านสถาปัตยกรรม ด้วยการแยก frontend และ backend ออกจากกัน ทำให้สามารถขยายแต่ละส่วนได้อิสระ และในส่วนของ backend มีการแยก service ตามโดเมนงาน คือ project, user, unit, budget-plan และ delegation เพื่อรองรับการเติบโตของฟีเจอร์ในอนาคตได้อย่างเป็นระบบ
3. ด้านการใช้งาน ได้มีการออกแบบ workflow ให้สอดคล้องกับการทำงานจริงของฝ่ายพัสดุ เช่น การมีระบบงานกลางที่เจ้าหน้าที่สามารถดึงงานเข้ามาในความรับผิดชอบของตนเอง การรองรับกระบวนการส่งงานกลับ เพื่อแก้ไขข้อมูล และหน้าติดตามสถานะสำหรับหน่วยงานผู้ขอ เพื่อลดการประสานงานผ่านเอกสาร โดยมีการออกแบบแผนภาพ project status state machine ดังแสดงในภาพที่ 3.5

### ภาพที่ 3.5 แผนภาพ project status state machine
![06-project-status-state-machine](https://github.com/capmoo-capstone/report/blob/main/report-diagrams/06-project-status-state-machine.png)

4. ด้านการติดตามงานและธรรมาภิบาล คณะผู้จัดทำออกแบบระบบมาให้ทุกขั้นตอนสำคัญของโครงการสามารถจัดเก็บสถานะและประวัติการแก้ไขในฐานข้อมูล ทำให้สามารถตรวจสอบย้อนหลังได้ 100% รองรับข้อกำกับด้านความโปร่งใสและธรรมาภิบาลของงานจัดซื้อจัดจ้างภาครัฐ
5. ด้านความสมบูรณ์ของข้อมูลและความเชื่อมั่นในระบบ ผ่านการกำหนด validation schema ด้วย Zod และกฎของข้อมูลในฐานข้อมูล เพื่อป้องกันข้อผิดพลาดตั้งแต่ก่อนบันทึกลงระบบ
6. ด้านความยั่งยืนและการดูแลระบบ มีการเลือกใช้เทคโนโลยีมาตรฐานที่มีชุมชนผู้พัฒนาสนับสนุนสูง เช่น React, Node.js, Express, PostgreSQL และออกแบบโครงสร้างโค้ดแบบ feature/service-driven เพื่อให้ทีมพัฒนาชุดต่อไปสามารถเข้ามาศึกษาและบำรุงรักษาระบบได้ง่าย

## 3.4 เปรียบเทียบข้อเสนอกับสิ่งที่พัฒนาจริง
ในส่วนนี้จะเป็นการเปรียบเทียบระหว่างข้อเสนอเริ่มต้น กับระบบสุดท้ายที่ได้ดำเนินการพัฒนาจริง ดังแสดงในตาราง ซึ่งเน้นให้เห็นถึงความครอบคลุมของระบบในระดับ Minimum Viable Product (MVP)

| ประเด็นการพัฒนาระบบ | แผนการพัฒนาที่เสนอ | สิ่งที่พัฒนาจริงในระบบ |
| :--- | :--- | :--- |
| **Authentication & Authorization** | รองรับการเข้าสู่ระบบแบบ RBAC พร้อมการจำกัดสิทธิ์การใช้งานของหน่วยงาน | พัฒนาระบบยืนยันตัวตนแบบครบวงจรด้วย JWT และ RBAC มีระบบการมอบหมายสิทธิ์แทน รวมทั้งการจำกัดโควตาผู้แทนหน่วยงานได้ |
| **Project Assignment** | สร้างระบบรับเรื่อง มีระบบงานกลาง ให้เจ้าหน้าที่เลือกรับงานเอง หรือรับมอบหมายจากหัวหน้างาน | มีระบบครบถ้วนตั้งแต่การสร้างโครงการ การนำเข้าข้อมูล การดึงงานจากระบบงานกลาง และการเปลี่ยนหรือเพิ่มผู้รับผิดชอบ |
| **Project Workflow** | แยกกระบวนการอนุมัติและสถานะการจัดซื้อตามวิธีจัดซื้อเฉพาะเจาะจง | มี lifecycle ที่ชัดเจน เช่น การอนุมัติ การเสนอ การลงนาม เป็นต้น โดยครอบคลุมตั้งแต่กระบวนการจัดซื้อไปจนถึงการบริหารสัญญาและการปิดโครงการ รวมทั้งมีระบบติดตามสถานะโครงการ |
| **Submission System** | รองรับการยื่นเอกสาร การส่งกลับงานไปแก้ไข หรือการยกเลิกโครงการ | ระบบรองรับการจัดการไฟล์เอกสารตามขั้นตอน มี endpoint สำหรับการส่งเรื่องแก้ไข (`/request-edit`) และการร้องขอยกเลิกซึ่งต้องผ่านกระบวนการอนุมัติ (`/approve-cancel`) |
| **Back Office Management** | ระบบจัดการข้อมูลเจ้าหน้าที่ กลุ่มงาน และหน่วยงานภายนอก | พัฒนาระบบจัดการข้อมูลเจ้าหน้าที่ กลุ่มมงาน หน่วยงานภายนอก พร้อมการตั้งผู้แทน และรองรับการนำเข้าข้อมูลแผนฯ เพื่อใช้เป็นฐานข้อมูลตั้งต้นของระบบ |

จากการเปรียบเทียบจะเห็นว่าระบบสุดท้ายที่ได้ดำเนินการพัฒนาจริงขยายขอบเขตจากเอกสารเสนอเดิมอย่างชัดเจน ระบบที่พัฒนาแล้วครอบคลุมแกนหลักด้านการรับเรื่อง มอบหมาย และติดตาม workflow ได้ครบถ้วน ซึ่งจะช่วยลดปัญหาคอขวดของการทำงาน และสร้างมาตรฐานกระบวนการใหม่ที่โปร่งใสตรวจสอบได้

# 4. ระเบียบวิธีพัฒนาโครงงาน (5-8 หน้า)

## 4.1 กระบวนการพัฒนา
คณะผู้จัดทำโครงงาน NexusProcure ได้นำแนวทางกระบวนการทำงานแบบ agile มาประยุกต์ใช้ โดยแบ่งการทำงานออกเป็น sprint เพื่อให้สามารถปรับปรุงและแก้ไขระบบได้อย่างยืดหยุ่นตามความต้องการของผู้ใช้งานจริง โดยกระบวนการพัฒนาแบ่งออกเป็นส่วนหลัก ๆ ดังนี้

1. การพัฒนาส่วน frontend
การพัฒนาส่วนนี้ใช้รูปแบบการจัดโครงสร้างแบบ Feature-Based Architecture เพื่อให้โค้ดมีความเป็นระเบียบและง่ายต่อการบำรุงรักษา โดยแบ่งโฟลเดอร์ตามฟังก์ชันการทำงานหลัก เช่น ฟังก์ชันการยืนยันตัวตน ฟังก์ชันการจัดการโครงการ ฟังก์ชันการอนุมัติ เป็นต้น ภายในแต่ละคุณสมบัติจะประกอบด้วยการแยกส่วนประกอบหน้าจอ และการจัดการข้อมูลผ่าน custom hooks เช่น การใช้ `useQuery` และ `useMutation` ของ TanStack Query เพื่อจัดการสถานะข้อมูลจาก server

2. การพัฒนาส่วน backend
มีการพัฒนา API และ business logic โดยอาศัย Layered Architecture เพื่อลดความซ้ำซ้อนและเพิ่มความปลอดภัย โดยโครงสร้างหลักประกอบด้วย
    - Routes Layer จัดการเส้นทางของ API Endpoint และระบุ middleware สำหรับการตรวจสอบสิทธิ์ด้วย JWT
    - Controllers Layer รับ request และทำการตรวจสอบความถูกต้องของข้อมูลผ่าน Zod พร้อมกับเรียกใช้งาน service
    - Services Layer รับผิดชอบ business logic ที่ซับซ้อน เช่น การสร้างโครงการ การจัดการวงจรสถานะโครงการ และการมอบหมายงาน โดยจะโต้ตอบกับฐานข้อมูลโดยตรงผ่าน Prisma ORM

3. การออกแบบฐานข้อมูล
มีการออกแบบและสร้างฐานข้อมูลด้วย Prisma Schema โดยจำลองโครงสร้างความสัมพันธ์ที่ซับซ้อนของฝ่ายการพัสดุ เช่น การจัดการโครงสร้างองค์กร บทบาทผู้ใช้ สิทธิ์การมอบหมายงานแทน และตารางบันทึกลำดับเหตุการณ์ ระบบมีการทำฐานข้อมูลจำลองเพื่อใช้ในการทดสอบระบบในช่วงการพัฒนา

## 4.2 เครื่องมือ เฟรมเวิร์ก และสภาพแวดล้อมในการพัฒนา
คณะผู้จัดทำพัฒนาระบบ NexusProcure โดยอาศัยเครื่องมือและเทคโนโลยีที่ทันสมัย เพื่อให้ระบบมีประสิทธิภาพ ปลอดภัย และสามารถดูแลรักษาได้ง่ายในระยะยาว ได้แก่

1. ภาษาเขียนโปรแกรม
    - TypeScript นำมาใช้เป็นภาษาหลักในการพัฒนาทั้งในส่วน frontend และส่วน backend เพื่อนำมาช่วยในเรื่องการตรวจสอบชนิดของตัวแปรก่อนการ run หรือ static typing ซึ่งช่วยลดข้อผิดพลาดในการเขียนโปรแกรมและทำให้โค้ดมีความน่าเชื่อถือ

2. ส่วน frontend
    - React และ Vite นำมาใช้ในการสร้างส่วนติดต่อผู้ใช้งานแบบ component-based และใช้ Vite เป็นเครื่องมือจัดการกระบวนการ build เพื่อความรวดเร็ว
    - Tailwind CSS และ shadcn/ui นำมาใช้สำหรับการจัดการสไตล์และส่วนประกอบหน้าจอ เพื่อให้ได้การออกแบบที่สวยงามและเป็นมาตรฐาน
    - TanStack Query นำมาใช้สำหรับการจัดการ cache และการสื่อสารข้อมูลกับ backend
    - TanStack Table นำมาใช้สำหรับการแสดงผลและจัดการตารางข้อมูลที่ซับซ้อน เช่น ตารางโครงการ และตารางระบบงานกลาง
    - React Hook Form และ Zod นำมาใช้สำหรับการจัดการฟอร์มและการตรวจสอบความถูกต้องของข้อมูลในฝั่ง client

3. ส่วน backend
    - Node.js และ Express นำมาใช้เป็นเฟรมเวิร์กหลักในการสร้าง RESTful API
    - Prisma ORM นำมาใช้เป็นเครื่องมือในการทำ Object-Relational Mapping เพื่อจัดการการเชื่อมต่อและดึงข้อมูลจากฐานข้อมูล PostgreSQL
    - JSON Web Token (JWT) นำมาใช้สำหรับการยืนยันตัวตน และการจัดการ session ผู้ใช้แบบ serverless
    - Zod นำมาใช้สำหรับการตรวจสอบรูปแบบและโครงสร้างข้อมูลฝั่ง server หรือ Payload Validation อย่างรัดกุมก่อนบันทึกลงฐานข้อมูล
    - Swagger UI นำมาใช้สำหรับสร้างหน้าเอกสารคู่มือ API เพื่อให้ทีมพัฒนาและทดสอบทำงานร่วมกันได้สะดวก

4. ระบบฐานข้อมูลและการจัดเก็บไฟล์
    - PostgreSQL ผ่านบริการ Neon Serverless นำมาใช้เป็นระบบ Relational Databaseหลักสำหรับเก็บข้อมูลของระบบทั้งหมด โดยเหตุผลที่คณะผู้จัดทำเลือกใช้สถาปัตยกรรมแบบ serverless แทนการติดตั้งแบบ self-hosted คือเพื่อตอบโจทย์บริบทของโครงงานในด้านความคุ้มค่า การบำรุงรักษาที่สะดวกรวดเร็วโดยไม่ต้องบริหารจัดการ infrastructure เอง และความสามารถในการขยายตัวอัตโนมัติตามปริมาณการใช้งาน
    - Cloudflare R2 นำมาใช้เป็นระบบจัดเก็บไฟล์แบบ object storage สำหรับจัดเก็บเอกสารประกอบโครงการ โดยทำงานแบบแยกส่วนจาก server หลักผ่านการใช้ Presigned URL เพื่อให้การอัปโหลดและดาวน์โหลดไฟล์มีความปลอดภัยและมีประสิทธิภาพ การเลือกใช้ Cloudflare R2 แทนการเก็บไฟล์บน server หลักหรือ self-hosted storage มีเหตุผลสำคัญคือช่วยลดภาระทรัพยากรของ server หลักได้อย่างมหาศาล อีกทั้ง Cloudflare R2 ยังมีข้อได้เปรียบด้านความคุ้มค่าสำหรับโครงงาน เนื่องจากไม่มีค่าใช้จ่ายในการดึงข้อมูลออก ซึ่งเหมาะสมอย่างยิ่งกับระบบงานพัสดุที่ต้องมีการดาวน์โหลดเอกสารเพื่อตรวจสอบอยู่เป็นประจำ

5. เครื่องมือจัดการโครงงานและการทำงานร่วมกัน
    - Git และ GitHub นำมาใช้สำหรับการทำ version control และทำงานร่วมกันในทีม
    - Figma นำมาใช้สำหรับการออกแบบ UI ก่อนนำมาเขียนโปรแกรมจริง
    - Notion นำมาใช้สำหรับการบริหารจัดการโครงการ บันทึกเอกสารต่าง ๆ และติดตามความคืบหน้าของงาน

เพื่อให้เห็นภาพรวมของโครงสร้างพื้นฐานและการนำเทคโนโลยีต่าง ๆ มาประกอบกันเพื่อใช้งานจริง คณะผู้จัดทำได้ออกแบบแผนภาพการ deploy ดังแสดงในภาพที่ 4.1

## 4.3 ปัญหาที่พบและการแก้ไข
ในระหว่างการพัฒนาโครงงาน NexusProcure คณะผู้จัดทำพบอุปสรรคทางเทคนิคต่าง ๆ ซึ่งคณะผู้จัดทำได้แก้ไขปัญหาดังกล่าวแล้วดังนี้

1. ความซับซ้อนในการจัดการสถานะโครงการและตรรกะของระบบงานกลาง
   - อุปสรรค: กระบวนการจัดซื้อจัดจ้างของมหาวิทยาลัยมีกฎเกณฑ์ที่ซับซ้อน เช่น การอนุญาตให้เจ้าหน้าที่เลือกโครงการได้ด้วยตนเอง การเปลี่ยนผู้รับผิดชอบ หรือการส่งกลับงานไปแก้ไข ซึ่งการดูแลสถานะเหล่านี้อาจทำให้เกิดข้อผิดพลาดได้ง่าย
   - แนวทางแก้ไข: คณะผู้จัดทำได้แยกการทำงานตรรกะโครงการออกเป็นโมดูลต่าง ๆ อย่างชัดเจน ได้แก่ `project-lifecycle.service.ts` สำหรับการเปลี่ยนสถานะโดยรวม และ `project-assignment.service.ts` สำหรับดูแลระบบงานกลาง นอกจากนี้ยังมีการบังคับเงื่อนไขสถานะ เพื่อป้องกันพฤติกรรมที่ไม่ถูกต้อง เช่น งานที่มีผู้รับผิดชอบแล้วจะไม่สามารถถูกเลือกซ้ำได้

2. การรักษาความถูกต้องของข้อมูลระหว่าง frontend และ backend
   - อุปสรรค: โครงการมีรูปแบบฟอร์มและการนำเข้าข้อมูลจำนวนมาก ทำให้การตรวจสอบความถูกต้องด้วยเจ้าหน้าที่ อาจส่งผลให้เกิดข้อผิดพลาดได้ง่าย และหากข้อมูลที่ไม่ถูกต้องถูกส่งไปยัง backend อาจทำให้เกิดปัญหาตามมา เช่น ข้อมูลขาดหาย หรือข้อมูลที่ไม่สอดคล้องกับโครงสร้างฐานข้อมูล
   - แนวทางแก้ไข: คณะผู้จัดทำประยุกต์ใช้ไลบรารี Zod เข้ามาเป็นตัวกลางในการตรวจสอบความถูกต้องของข้อมูล ทั้งฝั่ง client (ก่อนกดส่งฟอร์ม) และฝั่ง server (ก่อนนำเข้าสู่ service) ช่วยให้ระบบมีความปลอดภัยและป้องกันข้อมูลขยะได้อย่างมีประสิทธิภาพ

3. ความท้าทายในการจัดการแคชและลดการดึงข้อมูลที่ซ้ำซ้อนในฝั่ง frontend
   - อุปสรรค: เมื่อมีการอัปเดตข้อมูล เช่น การอนุมัติโครงการ ผู้ใช้มักไม่เห็นผลลัพธ์การเปลี่ยนแปลงในหน้าตารางโดยทันที เนื่องจากหน้าจอไม่ได้ดึงข้อมูลใหม่
   - แนวทางแก้ไข: มีการนำ TanStack Query เข้ามาบริหารจัดการแคชอย่างเป็นระบบ โดยการใช้ `queryClient.invalidateQueries` ร่วมกับการกำหนด query keys ที่เจาะจง ทำให้เมื่อมีการทำงานอะไรบางอย่างสำเร็จ ระบบจะทำการรีเฟรชข้อมูลเฉพาะส่วนที่เกี่ยวข้องเท่านั้น ส่งผลให้ข้อมูลมีความใหม่แบบ real-time โดยไม่เพิ่มภาระให้กับ server มากเกินไป

4. การจัดการสิทธิ์ผู้ใช้งานที่ซับซ้อนและการรับมอบอำนาจ
   - อุปสรรค: ผู้ใช้งาน 1 คนอาจมีหลายบทบาทในหลายหน่วยงาน หรือเป็นรักษาการแทนผู้บริหาร ซึ่งระบบต้องประมวลผลสิทธิ์เหล่านี้ในทุกครั้งที่มีการเรียกดูข้อมูล
   - แนวทางแก้ไข: สร้างฟังก์ชันส่วนกลาง คือ `permissions.ts` และ `user-role.ts` ควบคู่กับการใช้ Middleware กับฝั่ง server เพื่อตรวจเช็กสิทธิ์ และปรับปรุง query การค้นหา เพื่อจำกัดผลลัพธ์ข้อมูลให้ตรงกับขอบเขตความรับผิดชอบของบุคลากรรายนั้น ๆ โดยอัตโนมัติ

5. การตีความข้อกำหนดของระบบและการสื่อสารภายในทีม
   - อุปสรรค: ในช่วงเริ่มต้นของการพัฒนา พบว่าทีมผู้พัฒนาทำความเข้าใจรายละเอียดการทำงานของบางฟังก์ชันไม่ตรงกัน เนื่องจากการขาดเอกสาร Acceptance Criteria ที่ชัดเจน ทำให้การพัฒนาฟังก์ชันบางส่วนไม่ครอบคลุมตามความต้องการของผู้ใช้งาน
   - แนวทางแก้ไข: คณะผู้จัดทำได้ปรับปรุงกระบวนการทำงานแบบ agile โดยมอบหมายให้ Business Analyst รับผิดชอบในการจัดทำเอกสาร Acceptance Criteria อย่างละเอียดก่อนเริ่มการเขียนโค้ดในแต่ละ sprint เพื่อใช้เป็นมาตรฐานกลางในการเชื่อมโยงความเข้าใจระหว่าง requirement และกระบวนการพัฒนาระบบจริงให้มีความถูกต้องตรงกัน

6. การบริหารจัดการ bandwidth และภาระของ server ในการอัปโหลดไฟล์
   - อุปสรรค: ระบบมีการอัปโหลดและดาวน์โหลดเอกสารประกอบโครงการจำนวนมาก หากให้ผู้ใช้อัปโหลดไฟล์ผ่าน backend โดยตรง จะทำให้ server เกิดปัญหาคอขวดและใช้หน่วยความจำสูง
   - แนวทางแก้ไข: คณะผู้จัดทำเปลี่ยนสถาปัตยกรรมการอัปโหลดไปใช้ Presigned URL โดยให้ frontend ร้องขอ URL ชั่วคราวจาก backend จากนั้น frontend จะทำการอัปโหลดไฟล์ตรงไปยัง Cloudflare R2 เมื่อสำเร็จจึงส่ง key path กลับมาบันทึกที่ backend วิธีนี้ช่วยลดภาระการประมวลผลของ backend ได้อย่างมหาศาลและเพิ่มความรวดเร็วให้ผู้ใช้งาน

# 5. การทดสอบและการทดลองระบบ (3-5 หน้า)

## 5.1 แผนการทดสอบ/ทดลอง
การทดสอบระบบ NexusProcure มีวัตถุประสงค์เพื่อประเมินความถูกต้องของการทำงาน ความปลอดภัยในการเข้าถึงข้อมูล และความเสถียรของกระบวนการทำงานตาม requirement ของระบบ โดยมีการแบ่งแผนการทดสอบตามระดับความสำคัญ เพื่อให้มั่นใจว่าคณะผู้จัดทำจะตรวจสอบฟังก์ชันหลักที่ส่งผลกระทบต่อการทำงานของฝ่ายการพัสดุอย่างเข้มงวด ดังนี้

1. ระดับ P0 หรือกรณีเร่งด่วน เป็นการทดสอบที่ต้องผ่านเกณฑ์ทั้งหมดก่อนส่งมอบระบบ เน้นการตรวจสอบสิทธิ์การเข้าถึงข้อมูลของผู้ใช้งานตาม RBAC ความถูกต้องของ workflow state machine และความสมบูรณ์ของข้อมูลเมื่อมีการนำเข้าและบันทึกข้อมูล
2. ระดับ P1 หรือกรณีสำคัญ เป็นการทดสอบความถูกต้องของฟังก์ชันที่สนับสนุนการทำงาน เช่น การค้นหาหรือกรองข้อมูลขั้นสูง
3. ระดับ P2 หรือกรณีทั่วไป เป็นการทดสอบความเรียบร้อยของส่วน UI ความสะดวกในการใช้งาน และคุณภาพของการแสดงผลบนหน้าจอ

นอกจากนี้ แผนการทดสอบยังครอบคลุมวิธีการทดสอบสองรูปแบบ ได้แก่ การทดสอบแบบอัตโนมัติ สำหรับกรณีทดสอบที่เป็นฟังก์ชันเชิงตรรกะและเส้นทางการทำงานหลัก และการทดสอบด้วยบุคคล สำหรับกรณีที่ต้องการการตัดสินใจด้วยสายตา เช่น การตรวจสอบรูปแบบเอกสารที่ส่งออก หรือการแสดงผลบนหน้าจอขนาดต่าง ๆ

## 5.2 สภาพตั้งต้น
การเตรียมสภาพแวดล้อมและข้อมูลสำหรับการทดสอบระบบ มีการกำหนดปัจจัยและเงื่อนไขที่สะท้อนถึงการทำงานจริงของฝ่ายการพัสดุ ดังนี้

1. สภาพแวดล้อมทางฮาร์ดแวร์และซอฟต์แวร์
   - การทดสอบดำเนินการบน server จำลองที่ใช้สถาปัตยกรรมเดียวกับระบบจริง ได้แก่ Node.js (Express) สำหรับส่วนหลังบ้าน และ React สำหรับส่วนหน้าบ้าน
   - ฐานข้อมูลที่ใช้สำหรับการทดสอบคือ PostgreSQL ผ่านบริการ Neon Serverless
2. ชุดข้อมูลทดสอบ
   - มีการทำ script เพื่อสร้างข้อมูลจำลอง ซึ่งประกอบด้วยข้อมูลโครงสร้างองค์กรจำลอง (ของฝ่ายและกลุ่มงาน) และข้อมูลบัญชีผู้ใช้งานในบทบาทต่าง ๆ (ของผู้ดูแลระบบ ผู้แทนฝ่าย เจ้าหน้าที่พัสดุ หัวหน้ากลุ่มงาน ฯลฯ)
   - ชุดข้อมูลแผนงบประมาณจำลอง และไฟล์ข้อมูลโครงการรูปแบบ Microsoft Excel ที่จำลองมาจากรูปแบบระบบ LessPaper และ Fiori เพื่อทดสอบระบบการนำเข้าข้อมูลแบบกลุ่ม
3. เงื่อนไขการทดสอบ
   - มีการทดสอบภายใต้สิทธิ์การใช้งานที่แตกต่างกัน เพื่อยืนยันว่าผู้ใช้งานแต่ละบทบาทเห็นข้อมูลและมีสิทธิ์ในการดำเนินการต่าง ๆ เฉพาะในขอบเขตที่ตนเองรับผิดชอบเท่านั้น

## 5.3 ตัวชี้วัด
ในการประเมินประสิทธิภาพและความสมบูรณ์ของระบบ คณะผู้จัดทำได้กำหนดตัวชี้วัด สำหรับการทดสอบไว้ดังนี้

1. ด้านความถูกต้อง โดยอัตราความสำเร็จของการทำงานเทียบกับกรณีทดสอบทั้งหมด โดยเฉพาะในระดับ P0 ต้องมีอัตราการผ่านอยู่ที่ 100% ระบบต้องปฏิเสธการนำเข้าข้อมูลที่ไม่ถูกต้อง เช่น วงเงินติดลบ หรือเลขที่ใบขอซื้อขอจ้าง  หรือ Purchase Requisition (PR) ซ้ำซ้อนได้อย่างแม่นยำ
2. ด้านความปลอดภัยและสิทธิ์การเข้าถึง ผู้ใช้งานที่ไม่มีสิทธิ์ หรือไม่ได้เข้าสู่ระบบ ต้องไม่สามารถเข้าถึงหน้าจอและข้อมูลข้ามระดับการเข้าถึงของตนเองได้
3. ด้านความถูกต้องของสถานะ โครงการต้องสามารถเปลี่ยนสถานะตามลำดับที่กำหนดไว้ในวงจรสถานะการจัดซื้อจัดจ้าง โดยไม่เกิดการข้ามขั้นตอนที่ผิดกฎระเบียบ

## 5.4 กรณีทดสอบ
คณะผู้จัดทำได้จัดทำเอกสารกรณีทดสอบอย่างละเอียด ซึ่งครอบคลุมการทำงานในทุกฟังก์ชันหลักของระบบ โดยสามารถสรุปกรณีทดสอบที่สำคัญ ได้ดังตัวอย่างต่อไปนี้

1. ระบบการนำเข้าข้อมูลโครงการ (Import Project Page)
    - ทดสอบระบบการตรวจสอบสิทธิ์ ให้มั่นใจว่ามีเพียงผู้ดูแลระบบ หรือผู้แทนหน่วยงานที่มีสิทธิ์เท่านั้นที่สามารถเข้าถึงหน้านำเข้าโครงการได้ หากผู้ใช้งานไม่มีสิทธิ์ ระบบจะทำการปฏิเสธการเข้าถึงและแสดงข้อความแจ้งเตือนที่เหมาะสม
    - ทดสอบระบบการนำเข้าข้อมูลแบบกลุ่มผ่านการอัปโหลดไฟล์ Microsoft Excel ทั้งรูปแบบของไฟล์ LessPaper และ Fiori โดยระบบต้องสามารถตรวจสอบความซ้ำซ้อนของเลขที่ใบขอซื้อขอจ้าง (PR) ตรวจสอบความถูกต้องของวงเงินงบประมาณ (ต้องเป็นตัวเลขและมากกว่า 0) และตรวจสอบวันที่ส่งมอบงาน (ต้องเป็นวันที่ในอนาคต)
    - ทดสอบตรรกะการเลือกแผนงบประมาณ โดยในกรณีที่มีการเลือกแผนงบประมาณหลายรายการ ระบบต้องคำนวณและแสดงยอดรวมงบประมาณเริ่มต้นอัตโนมัติ และแจ้งเตือนหากผู้ใช้งานระบุยอดเงินที่น้อยกว่าผลรวมของแผน
    - ทดสอบการเชื่อมโยงข้อมูล ผ่านการตรวจสอบว่าเมื่อนำเข้าข้อมูลสำเร็จแล้ว ระบบมีการสร้างเลขที่ลงรับ คำนวณระดับความเร่งด่วน จัดโครงการเข้าสู่ระบบงานกลาง ของกลุ่มงานที่ถูกต้อง และกำหนดสถานะเป็น ยังไม่ได้มอบหมาย หรือไม่

2. ระบบการนำเข้าแผนรายงานจัดซื้อจัดจ้าง ครุภัณฑ์ และสิ่งก่อสร้างประจำปีงบประมาณ (Import Budget Plan Page)
    - ทดสอบว่ามีเพียงเฉพาะผู้ดูแลระบบเท่านั้น ที่สามารถเข้าใช้งานระบบนี้ได้ ผู้ใช้บทบาทอื่นจะไม่สามารถมองเห็นปุ่มหรือเข้าถึงหน้าต่างนำเข้าได้
    - ทดสอบการตรวจสอบรูปแบบข้อมูลของคอลัมน์ที่จำเป็นทั้งหมด เช่น ปีงบประมาณ ศูนย์ต้นทุน หน่วยงาน เงินทุน และวงเงินงบประมาณ หากข้อมูลไม่ครบถ้วน ระบบจะไม่อนุญาตให้บันทึกและแจ้งเตือนแถวที่มีข้อผิดพลาดให้แก้ไขในตาราง

3. ระบบจัดการและติดตามโครงการ (All Projects Page)
    - ทดสอบว่าเจ้าหน้าที่พัสดุสามารถเห็นโครงการทั้งหมดจากทุกหน่วยงานได้ ในขณะที่ผู้แทนหน่วยงานจะมองเห็นเฉพาะโครงการของหน่วยงานตนเองเท่านั้น
    - ทดสอบระบบการค้นหาแบบ global search ด้วยเลขที่ลงรับ ชื่อโครงการ และชื่อผู้รับผิดชอบ รวมถึงทดสอบตัวกรองข้อมูลขั้นสูง เช่น การกรองตามสถานะ ช่วงเวลา วิธีการจัดหา และหน่วยงาน โดยการเปลี่ยนแปลงของข้อมูลในตารางต้องตอบสนองแบบ real-time
    - ทดสอบการแสดงผลของแท็กและสีของสถานะโครงการ เช่น ยังไม่ได้มอบหมาย (สีเทา) รอการตอบรับ (สีเหลือง) เสร็จสิ้น (สีเขียว) และ ยกเลิก (สีแดง) ซึ่งต้องแสดงผลถูกต้องและแตกต่างกันตามบทบาทของผู้ใช้งาน

จากผลการทดสอบเบื้องต้นตามแผนการทดสอบ พบว่าระบบสามารถรองรับหลักการตรวจสอบสิทธิ์ และการจัดการ project state machine ตามเป้าหมายที่กำหนดไว้ได้อย่างครบถ้วนและถูกต้อง

# 6. ผลการดำเนินงาน (3-5 หน้า)

## 6.1 ผลลัพธ์
จากการประเมินผลการทดสอบระบบ NexusProcure ตามกรณีทดสอบที่ได้ระบุไว้ในบทก่อนหน้า คณะผู้จัดทำได้ทำการทดสอบระบบกับฟังก์ชันการทำงานหลักในระดับ MVP โดยครอบคลุมทั้งกระบวนการตรวจสอบสิทธิ์ กระบวนการทำงาน และความถูกต้องของข้อมูล ซึ่งมีจำนวนกรณีทดสอบที่นำมาประเมินทั้งสิ้น 350 กรณี แบ่งระดับความสำคัญออกเป็น ระดับ P0 P1 และ P2 มีผลการทดสอบดังนี้

## 6.2 วิเคราะห์และตีความผล
จากผลการทดสอบและการประเมินระบบเมื่อเทียบกับวัตถุประสงค์ของโครงงาน สามารถวิเคราะห์และตีความผลลัพธ์ได้ดังนี้

1. ประสิทธิภาพในการติดตามและบริหารจัดการงานพัสดุ
ระบบ NexusProcure สามารถทำหน้าที่เป็นศูนย์กลางในการติดตามสถานะกระบวนการจัดซื้อจัดจ้างได้อย่างเป็นรูปธรรม การที่กรณีทดสอบระดับ P0 ทั้งหมดผ่านเกณฑ์ เป็นเครื่องยืนยันว่าระบบสามารถเก็บบันทึกและอัปเดตสถานะของโครงการ ตั้งแต่การออกเลขที่ลงรับ การเปลี่ยนช่วงการดำเนินงาน ไปจนถึงการปิดโครงการ ได้อย่างถูกต้องตามระเบียบการจัดซื้อจัดจ้าง ผู้ใช้งานสามารถเข้าถึงข้อมูลของโครงการที่ตนเองรับผิดชอบได้แบบ real-time ทำให้สามารถทดแทนการทำงานที่พึ่งพาเอกสารกระดาษในขั้นตอนการเตรียมการ และการประสานงานภายในได้อย่างมีประสิทธิภาพ

2. การลดปัญหาคอขวดและเพิ่มความโปร่งใสในการกระจายงาน
การนำระบบงานกลางมาใช้เป็นปัจจัยสำคัญที่ช่วยแก้ปัญหาความไม่ชัดเจนของผู้รับผิดชอบ จากการทดสอบระบบการมอบหมายงาน พบว่าระบบสามารถบังคับใช้กฎเกณฑ์การกระจายงานได้อย่างเข้มงวด เช่น การจำกัดจำนวนผู้รับผิดชอบไม่เกิน 2 คนต่อขั้นตอน และการแสดงชื่อผู้ปฏิบัติงานพร้อมวันที่รับงานในประวัติการแก้ไข สิ่งนี้ไม่เพียงแต่ช่วยลดปัญหาปริมาณงานสะสม แต่ยังช่วยให้ผู้บริหารสามารถตรวจสอบย้อนหลังและระบุขั้นตอนที่เป็นคอขวดได้อย่างแม่นยำ

3. การลดข้อผิดพลาดและเพิ่มความรวดเร็วในการนำเข้าข้อมูล
การประยุกต์ใช้ Zod สำหรับการตรวจสอบความถูกต้องของข้อมูลทั้งฝั่งหน้าบ้านและหลังบ้าน ช่วยป้องกันข้อผิดพลาดจากการกรอกข้อมูลของมนุษย์ได้อย่างมีนัยสำคัญ โดยผลการทดสอบยืนยันว่าระบบสามารถดักจับการนำเข้าข้อมูลที่ไม่ถูกต้อง เช่น วงเงินงบประมาณที่ติดลบ วันที่ส่งมอบงานในอดีต หรือเลขที่ใบขอซื้อขอจ้างที่ซ้ำซ้อน และแจ้งเตือนผู้ใช้งานให้แก้ไขก่อนนำเข้าสู่ฐานข้อมูล ทำให้ข้อมูลมีความน่าเชื่อถือและลดภาระในการตามแก้ไขข้อมูลในภายหลัง

## 6.3 ข้อจำกัดและสิ่งที่ได้เรียนรู้
แม้ว่าระบบจะสามารถทำงานได้ตามเป้าหมายหลัก แต่ในกระบวนการพัฒนาและการทดสอบ คณะผู้จัดทำได้พบข้อจำกัดและได้รับบทเรียนที่สำคัญ ดังนี้

1. ข้อจำกัดด้านประสิทธิภาพการค้นหาและกรองข้อมูล
ในการพัฒนาฟังก์ชันการค้นหา และกรองข้อมูลขั้นสูงที่ต้องดึงข้อมูลจากตารางที่มีความสัมพันธ์กันหลายชั้น ผ่าน Prisma ORM พบว่าเมื่อเงื่อนไขการค้นหามีความซับซ้อนมากขึ้น ส่งผลให้การสร้าง query อาจไม่ครอบคลุมเงื่อนไขบางประการ ทำให้ผลลัพธ์การค้นหาคลาดเคลื่อนดังที่พบในกรณีทดสอบระดับ P1 บทเรียนที่ได้รับคือ ในการพัฒนาระบบที่มีความซับซ้อนของข้อมูลสูง ทีมผู้พัฒนาจำเป็นต้องให้ความสำคัญกับการออกแบบ index และการเขียน raw query ในบางจุดที่ ORM ไม่สามารถประมวลผลได้อย่างมีประสิทธิภาพสูงสุด

2. ความท้าทายในการจัดการสถานะฝั่ง client
การใช้เทคโนโลยีการพัฒนาแบบ component-based ร่วมกับ TanStack Query ช่วยให้ระบบทำงานได้อย่างรวดเร็ว แต่ก็นำมาซึ่งความท้าทายในการจัดการ cache โดยคณะผู้จัดทำพบว่าปัญหา bug ในช่วงแรกของการพัฒนา มักเกิดจากการที่หน้าจอไม่ยอมดึงข้อมูลใหม่หลังจากที่ผู้ใช้ได้ดำเนินการอะไรบางอย่างสำเร็จ ทำให้ผู้ใช้เห็นข้อมูลเก่า บทเรียนที่ได้รับคือ การกำหนดโครงสร้างของ query keys ให้เป็นระบบและมีลำดับชั้นที่ชัดเจน เป็นหัวใจสำคัญที่จะช่วยให้การสั่งล้าง cache ทำงานได้อย่างถูกต้องและแม่นยำ

3. ความซับซ้อนของกฎระเบียบและข้อยกเว้นในกระบวนการจริง
ถึงแม้คณะผู้จัดทำจะพยายามจำลอง state machine ให้สอดคล้องกับระเบียบการจัดซื้อจัดจ้างมากที่สุด แต่ในระหว่างการวิเคราะห์ระบบพบว่ามี edge cases จำนวนมากในกระบวนการทำงานจริง เช่น โครงการที่ต้องรอการอนุมัติข้ามหน่วยงาน หรือโครงการที่มีความเร่งด่วนพิเศษที่ต้องข้ามขั้นตอนบางประการ ซึ่งการเขียนเงื่อนไขรองรับข้อยกเว้นทั้งหมดจะทำให้โค้ดมีความซับซ้อนและดูแลรักษายาก บทเรียนที่ได้รับคือ การพัฒนาระบบสำหรับหน่วยงานภาครัฐ จำเป็นต้องหาจุดสมดุลระหว่างการบังคับใช้กฎระเบียบอย่างเข้มงวด กับการอนุญาตให้มีความยืดหยุ่นในการแก้ไขข้อมูล เพื่อไม่ให้ระบบกลายเป็นอุปสรรคต่อการทำงานของเจ้าหน้าที่เสียเอง

# 8. จริยธรรม ความเป็นส่วนตัว และความเป็นมืออาชีพ (1 หน้า)

## 8.1 ประเด็นทางจริยธรรม
คณะผู้จัดทำให้ความสำคัญกับความเป็นส่วนตัว และความเป็นธรรมในการปฏิบัติงานเป็นอย่างมาก ในด้านความเป็นส่วนตัว ระบบมีการจัดการสิทธิ์การเข้าถึงข้อมูลที่รัดกุมผ่านระบบยืนยันตัวตนแบบ RBAC เพื่อให้มั่นใจว่าข้อมูลการจัดซื้อจัดจ้างและข้อมูลของร้านค้าผู้เสนอราคาจะถูกเข้าถึงได้เฉพาะผู้ที่มีอำนาจหน้าที่เกี่ยวข้องเท่านั้น ในส่วนของความเป็นธรรม มีระบบคิวงานกลางที่ถูกออกแบบมาเพื่อกระจายภาระงานให้แก่เจ้าหน้าที่พัสดุอย่างสมดุล ช่วยให้การจ่ายงานมีความโปร่งใส สามารถตรวจสอบปริมาณงานของเจ้าหน้าที่แต่ละท่านเพื่อป้องกันความเหลื่อมล้ำในการรับผิดชอบงานได้อย่างมีประสิทธิภาพ

## 8.2 กฎหมายที่เกี่ยวข้อง
เนื่องจากระบบนี้เกี่ยวข้องกับข้อมูลส่วนบุคคลต่าง ๆ โดยตรง ทำให้ต้องดำเนินงานการจัดเก็บและประมวลผลข้อมูลให้สอดคล้องกับพระราชบัญญัติคุ้มครองข้อมูลส่วนบุคคล พ.ศ. 2562 อีกทั้งต้องออกแบบกระบวนการทำงานภายในระบบให้สอดคล้องกับพระราชบัญญัติการจัดซื้อจัดจ้างและการบริหารพัสดุภาครัฐ พ.ศ. 2560 เพื่อเสริมสร้างความโปร่งใสตามมาตรฐานการดำเนินงานของหน่วยงานภาครัฐ

## 8.3 ผลกระทบต่อสิ่งแวดล้อม
ระบบดังกล่าวจะมีส่วนช่วยอย่างมากในการสร้างความยั่งยืนด้านสิ่งแวดล้อม โดยการเปลี่ยนผ่านกระบวนการทำงานที่พึ่งพากระดาษ ไปสู่รูปแบบดิจิทัล ช่วยลดการใช้ทรัพยากรกระดาษ ลดปริมาณขยะ และลดการปล่อยก๊าซเรือนกระจกที่เกิดจากการเดินทางเพื่อจัดส่งเอกสารทางกายภาพระหว่างหน่วยงานภายในมหาวิทยาลัยได้อย่างมีนัยสำคัญ

## 8.4 สิทธิ์ในทรัพย์สินทางปัญญา
สิทธิ์ในทรัพย์สินทางปัญญาของ source code ฐานข้อมูล และองค์ประกอบต่าง ๆ ของระบบ NexusProcure ถือเป็นลิขสิทธิ์ของสำนักบริหารการเงิน การบัญชี และการพัสดุ (สบง.) จุฬาลงกรณ์มหาวิทยาลัย ในฐานะผู้ว่าจ้างและเจ้าของโครงการแต่เพียงผู้เดียว ในส่วนของการนำเทคโนโลยีและเครื่องมือแบบ open-source หรือ third-party libraries มาใช้งาน คณะผู้จัดทำได้ปฏิบัติตามเงื่อนไขของสัญญาอนุญาต ที่เกี่ยวข้องอย่างเคร่งครัด เพื่อป้องกันการละเมิดลิขสิทธิ์

# 9. ความเสี่ยงและการจัดการทรัพยากร (1 หน้า)

## 9.1 ความเสี่ยง
ในการพัฒนาระบบ NexusProcure คณะผู้จัดทำได้ประเมินความเสี่ยงหลักที่อาจส่งผลกระทบและกำหนดแนวทางป้องกันไว้ ได้แก่ 1) ความเสี่ยงด้านความซับซ้อนของข้อมูลและตรรกะระบบ เนื่องจากกระบวนการจัดซื้อจัดจ้างมีเงื่อนไขและสถานะการทำงานที่ซับซ้อน หากการจัดการสถานะผิดพลาดอาจทำให้กระบวนการทำงานหยุดชะงัก เพื่อแก้ไขความเสี่ยงดังกล่าว คณะผู้จัดทำได้ออกแบบสถาปัตยกรรมระบบโดยแยกโมดูลการทำงานอย่างชัดเจนและใช้เครื่องมือตรวจสอบประเภทข้อมูล หรือ type checking อย่างรัดกุม และ 2) ความเสี่ยงด้านการยอมรับของผู้ใช้งาน เนื่องจากเจ้าหน้าที่พัสดุมีความคุ้นเคยกับระบบกระดาษ และระบบเดิมอาจเกิดความกังวลหรือต้านทานในการเปลี่ยนไปใช้ระบบดิจิทัล เพื่อแก้ไขความเสี่ยงดังกล่าว คณะผู้จัดทำจึงเน้นการออกแบบส่วน UI ให้มีความเป็นมิตร ใช้งานง่ายและมีขั้นตอนที่สอดคล้องกับกระบวนการทำงานจริง

## 9.2 การปรับตัว
ในระหว่างการดำเนินงาน คณะผู้จัดทำพบอุปสรรคด้านความคลาดเคลื่อนในการทำความเข้าใจความต้องการของระบบระหว่างสมาชิกในทีมผู้พัฒนา จึงได้มีการปรับเปลี่ยนกระบวนการทำงานแบบ agile โดยมอบหมายให้ Business Analyst จัดทำ Acceptance Criteria อย่างละเอียดก่อนเริ่มการพัฒนาในแต่ละ sprint เพื่อใช้เป็นมาตรฐานกลางในการสื่อสารและควบคุมคุณภาพของระบบให้ตรงตามความต้องการอย่างแท้จริง

## 9.3 ทรัพยากร
การพัฒนาโครงงานนี้อาศัยทรัพยากรต่าง ๆ ดังนี้ 1) ด้านซอฟต์แวร์ โดยพัฒนาส่วน frontend ด้วย React (Vite), TypeScript, Tailwind CSS และ shadcn/ui ส่วน backend ใช้ Node.js (Express) ร่วมกับ Prisma ORM และฐานข้อมูล PostgreSQL 2) ด้าน infrastructure โดยใช้คอมพิวเตอร์ส่วนบุคคลของทีมงานสำหรับการพัฒนาเบื้องต้น และ 3) ด้านข้อมูลทดสอบ คณะผู้จัดทำใช้ข้อมูลจำลอง และไฟล์ Microsoft Excel แผนงบประมาณจำลอง เพื่อทดสอบหลักการทำงานและฟังก์ชันการนำเข้าข้อมูลแบบกลุ่ม

## 9.4 การพิจารณาค่าใช้จ่าย
โครงงานนี้มีการบริหารจัดการงบประมาณอย่างรัดกุม โดยเน้นการใช้ประโยชน์จากเครื่องมือแบบ open-source ทำให้มีต้นทุนในด้านซอฟต์แวร์การพัฒนาที่ต่ำมาก ซึ่งคาดการณ์ว่าค่าใช้จ่ายรวมตลอดระยะเวลาจะอยู่ในเกณฑ์ที่ต่ำและไม่เกินขอบเขตงบประมาณที่กำหนด

# 10. สรุปและอภิปรายผลการดำเนินงาน (1-2 หน้า)

## 10.1 ผลสัมฤทธิ์
โครงงาน NexusProcure ประสบความสำเร็จในการพัฒนาระบบบริหารจัดการและติดตามสถานะกระบวนการจัดซื้อจัดจ้างสำหรับฝ่ายการพัสดุ สบง. โดยระบบดังกล่าวสามารถทำหน้าที่เป็นศูนย์กลางในการรวบรวมข้อมูลโครงการ เอกสารแนบ และติดตามสถานะการดำเนินงานในรูปแบบ real-time ซึ่งเป็นการแก้ไขปัญหาความล่าช้าและปัญหาคอขวดที่เกิดจากการทำงานด้วยเอกสารกระดาษได้อย่างเป็นรูปธรรม มีการพัฒนาระบบงานกลาง พร้อมทั้งบูรณาการระบบควบคุมสิทธิ์การเข้าถึงข้อมูลตามบทบาท ซึ่งผลการทดสอบยืนยันว่าระบบทำงานได้อย่างถูกต้องและมีประสิทธิภาพตามเป้าหมาย

เมื่อพิจารณาเชิงวิเคราะห์ร่วมกับช่องโหว่ของระบบเดิมในบทที่ 2 พบว่าโครงงานนี้สามารถปิดช่องโหว่ได้อย่างเป็นรูปธรรม ดังนี้ ประการแรก การปิดช่องโหว่ด้านการตรวจสอบเอกสาร คณะผู้จัดทำพัฒนาระบบให้สามารถรวบรวมข้อมูลและเอกสารจากทุกขั้นตอนมาแสดงผลรวมกัน ทำให้ผู้บริหารตรวจสอบความถูกต้องเพื่อประกอบการตัดสินใจได้อย่างรวดเร็ว ประการที่สอง การปิดช่องโหว่ด้านความครอบคลุมของวงจรจัดซื้อ ระบบรองรับกระบวนการทำงานแบบครบวงจร ตั้งแต่การนำเข้าแผนงบประมาณจนถึงการบริหารสัญญา พร้อมทั้งพัฒนาระบบรองรับบุคคลภายนอก ที่เชื่อมโยงผู้มีส่วนได้ส่วนเสียทุกฝ่ายเข้าด้วยกันอย่างสมบูรณ์ และ ประการสุดท้าย การปิดช่องโหว่ด้านการติดตามสถานะและปัญหาคอขวด คณะผู้จัดทำนำระบบงานกลางมาประยุกต์ใช้เพื่อกระจายงานอย่างเป็นธรรม รวมทั้งมีระบบบันทึก audit logs แบบอัตโนมัติ ทำให้สามารถระบุและแก้ไขปัญหาคอขวดแฝงที่เกิดจากการทำงานด้วยเอกสารกระดาษได้อย่างแม่นยำ นอกจากนี้การบูรณาการระบบควบคุมสิทธิ์การเข้าถึงข้อมูลตามบทบาท (RBAC) ยังช่วยยกระดับความปลอดภัย ซึ่งผลการทดสอบยืนยันว่าระบบสามารถทำงานเพื่อปิดช่องโหว่เหล่านี้ได้อย่างมีประสิทธิภาพตามเป้าหมาย

## 10.2 ความสอดคล้องกับวัตถุประสงค์โครงงาน
ระบบที่พัฒนาขึ้นสามารถตอบสนองวัตถุประสงค์ของโครงงานได้อย่างครบถ้วน ดังนี้ 1) ระบบสามารถรวบรวมและแสดงสถานะกระบวนการจัดซื้อจัดจ้างในแต่ละขั้นตอนได้อย่างชัดเจน ทดแทนระบบกระดาษแบบเดิมได้เกือบสมบูรณ์ 2) ระบบมีการบันทึก history logs ที่สามารถระบุชื่อเจ้าหน้าที่ผู้รับผิดชอบและเวลาดำเนินการ และ 3) ระบบสามารถป้องกันการนำเข้าข้อมูลที่ผิดพลาด ลดความซ้ำซ้อน และเพิ่มความสะดวกรวดเร็วในการประสานงานระหว่างหน่วยงานภายในมหาวิทยาลัยได้อย่างมีประสิทธิภาพ

## 10.3 การนำไปใช้
ระบบ NexusProcure มีศักยภาพสูงในการนำไปใช้งานจริงภายในจุฬาลงกรณ์มหาวิทยาลัย เนื่องจากคณะผู้จัดทำออกแบบสถาปัตยกรรมระบบมาให้รองรับผู้ใช้งานจำนวนมาก และสามารถขยายขนาดได้ในอนาคต อีกทั้งโครงสร้างของระบบและรูปแบบ business logic ยังสามารถนำไปปรับใช้เป็น prototype สำหรับหน่วยงานภาครัฐหรือสถาบันการศึกษาอื่น ๆ ที่ต้องการพัฒนาระบบ e-Procurement เพื่อยกระดับความโปร่งใสและประสิทธิภาพการทำงานภายในองค์กรของตนเองได้อีกด้วย

## 10.4 งานในอนาคต
คณะผู้จัดทำขอนำเสนอแนวทางสำหรับการพัฒนาโครงงานในอนาคต ดังนี้ 1) พัฒนาการเชื่อมต่อระบบเข้ากับฐานข้อมูลของมหาวิทยาลัย เพื่อให้บุคลากรสามารถใช้บัญชี CU Net เข้าสู่ระบบได้โดยตรง 2) พัฒนาฟังก์ชันแจ้งเตือนผู้รับผิดชอบทันทีผ่านช่องทาง email หรือหน้าเว็บไซต์ เพื่อให้เจ้าหน้าที่พัสดุสามารถติดตามงานได้อย่างทันท่วงทีและลดความเสี่ยงในการล่าช้า และ 3) พัฒนาระบบให้มีการนำเสนอข้อมูลบน dashboard เพื่อช่วยให้ผู้บริหารสามารถประเมินประสิทธิภาพของเจ้าหน้าที่พัสดุแต่ละคนได้อย่างแม่นยำ และสามารถวางแผนการปรับปรุงกระบวนการทำงานได้อย่างมีประสิทธิภาพมากขึ้น
