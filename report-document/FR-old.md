# Functional Requirements Specification (FRS)

Project Name: Nexus Procure (MVP)

## 1. ระบบยืนยันตัวตนและจัดการผู้ใช้งาน

### 1.1 Internal User Login (CU Staff & Units)

- **LDAP Integration:** ยืนยันตัวตนผ่าน **CU Net (LDAP)**
- **Unit Representative (Authorized User):**
    - **Constraint:** จำกัดสิทธิ์ให้มี **ผู้แทนหน่วยงานเพียง 1 คน** ต่อ 1 หน่วยงาน/ภาควิชา ที่สามารถสร้างโครงการ (Direct Input) ได้
    - **Admin Configuration:** Admin ส่วนกลางมีหน้าจอสำหรับจับคู่ (Map) `หน่วยงาน` <-> `User Account` (หากมีการเปลี่ยนคน Admin ต้องเป็นคนแก้ให้)
- **Staff/Head:** เจ้าหน้าที่พัสดุและหัวหน้างาน เข้าใช้งานตาม Role ปกติ (กำหนดสิทธิ์โดย Admin)

### 1.2 External User Interaction (Vendor)

- **No Login Required:** ผู้ขาย**ไม่ต้องล็อกอิน**เข้าสู่ระบบ
- **Submission Channel:** มีหน้าเว็บสาธารณะ (Public Web Form) สำหรับวางบิล
- **Validation Mechanism:**
    - ผู้ขายต้องระบุ **เลขที่ใบสั่งซื้อ (PO Number)** (จำเป็นต้องกรอก)
    - ระบบตรวจสอบว่ามีเลข PO นี้จริงในระบบหรือไม่ (เพื่อป้องกัน Spam) แต่จะไม่แสดงข้อมูลเชิงลึกของโครงการเพื่อความปลอดภัย
    - หากเลข PO ถูกต้อง → อนุญาตให้อัปโหลดไฟล์ใบเสร็จ/หลักฐานการส่งมอบ
- **Security:** ใช้ CAPTCHA หรือ Rate Limiting ป้องกันการยิง Spam

## 2. ระบบการรับเรื่องและมอบหมายงาน

### 2.1 Job Creation (นำเข้าโครงการ)

- **Methods:**
    1. **Direct Input:** หน่วยงานกรอกข้อมูลเอง → ระบบ Gen "เลขรับ (Receive No.)" ของระบบเราให้ทันที
    2. **Admin Input:** อัพโหลด Excel จาก Less Paper/Fiori มา Key เข้าระบบแทน (สนับสนุนให้ User มากรอกเองในอนาคต)
- **Data Fields:**
    - ข้อมูลพื้นฐาน (ชื่อโครงการ, งบประมาณ)
    - เลข PR (จาก SAP - ถ้ามี)
    - แนบไฟล์ (Optional)
- **Visibility:** User หน่วยงานภายนอก (ผู้ขอ) เห็นเฉพาะ Timeline, สถานะ, และชื่อผู้รับผิดชอบ (ไม่เห็นไฟล์ภายใน หรือรายละเอียดการตีกลับ internal)

### 2.2 Assignment Logic (การแจกจ่ายงาน)

- **Pool System:** งานใหม่จะถูกส่งเข้า "กลุ่มงาน (Team Pool)" ของแต่ละกลุ่มงาน
- **Self-Picking (Shopping):** เจ้าหน้าที่ ในกลุ่มนั้น สามารถกด "เลือกงาน (Pick Job)" เข้ามาเป็นของตัวเองได้
- **Manual Assign:** หากไม่มีใครเลือก หัวหน้าสามารถกด Assign ระบุตัวบุคคลได้ คนที่ถูก assign ต้องมากด confirm ก่อนรับงานนั้น ๆ
- **Job Distribution View:**  เพื่อดูว่าเจ้าหน้าที่คนไหนมีเท่าไหร่ (เพื่อเกลี่ยงานให้เหมาะสม)

## 3. Procurement Workflow (กระบวนการจัดซื้อ)

ระบบแยก Flow ตามวิธีจัดซื้อ ดังนี้:

### 3.1 วิธีเฉพาะเจาะจง (Specific Method: >100k - 500k)

- **Concurrent Steps:** ขั้นตอนที่ 1, 2, 3 สามารถทำ **พร้อมกันได้** (Parallel) ไม่ต้องรอตามลำดับ (ติ๊กจบเมื่อไหร่ก็ได้) แต่ต้องเสร็จครบก่อนไปขั้นถัดไป
- **Contract Number Gen:** ระบบต้อง Generate เลขที่สัญญาได้ (Format: เลขที่/ปีงบ เช่น 1/69 เริ่มรัน 1 ต.ค.) และดึงเลขหลักประกันสัญญาได้
- **Approval Loop:** เจ้าหน้าที่เสนอ → หัวหน้า → ผอ. (Flow ส่งไป-กลับจนกว่าจะอนุมัติ)
- **Vendor Info:** เจ้าหน้าที่กรอกข้อมูล Vendor (ชื่อ, Tax ID, Email) และเลข PO (จาก SAP) ในขั้นตอนนี้

### 3.2 วิธี E-bidding

- **Sequential Logic:** บังคับทำตามลำดับ (เช่น ต้องเสร็จข้อ 7 ทำ PR ถึงจะไปทำข้อ 8 ร่างสัญญาได้)
- **Additional Steps:**
    - บันทึกวันประชุม/รายชื่อคณะกรรมการ TOR
    - แนบเอกสารบังคับ 5 รายการ
    - (Future Phase) ระบบจับเวลาวิจารณ์ 7 วัน
    - บันทึก MIGO (เลขตรวจรับใน ERP) ในขั้นตอนสุดท้าย

### 3.3 Executive Review (One-Click)

- ผู้บริหาร (ผอ./หัวหน้า) ต้องมีหน้าจอ **"Summary View"** ที่รวมเอกสารแนบจากทุก Step (1-N) มาแสดงในหน้าเดียว เพื่อกดอนุมัติครั้งเดียว (ไม่ต้องไล่กดดูทีละ Step)

### 3.4 Timeline

- **Export:** สามารถ Export Timeline ของแต่ละโครงการเป็น **PDF** (จัด Format ให้สวยงามสำหรับการรายงาน)

## 4. Contract Management & Inspection (บริหารสัญญาและตรวจรับ)

### 4.1 Notifications & Billing (การแจ้งเตือนและวางบิล)

- **Email Alert:** ส่งอีเมลแจ้งเตือน Vendor และ กรรมการตรวจรับ (ข้อจำกัด: ส่งได้สูงสุด 17 คน/batch)
- **Vendor Submission:** Vendor กรอกฟอร์มส่งงาน/วางบิล Online (หรือส่งเอกสารให้ธุรการกรอกให้)
- **Document Check:** เจ้าหน้าที่พัสดุตรวจสอบเอกสารวางบิล → ถ้ารับเรื่อง → ระบบแจ้งเตือนกรรมการตรวจรับ (Option: ให้เจ้าหน้าที่กดปุ่ม "แจ้งเตือน" เองได้เพื่อความชัวร์)

### 4.2 Inspection & Finance Handover (ตรวจรับ-การเงิน-ทะเบียน)

- **Inspection Recording:** บันทึกวันที่ตรวจรับ, วันที่ประชุม, ผลการตรวจรับ
- **MIGO Input:** กรอกเลขที่ใบรับของ (MIGO) จากระบบ SAP เพื่ออ้างอิง
- **Finance Integration:**
    - หัวหน้าอนุมัติ → ส่งสถานะไป "ฝ่ายการเงิน"
    - **Return Flow:** การเงินสามารถ "ส่งคืน (Return)" ให้พัสดุแก้ไขได้ (เช่น ใบเสร็จผิด) ระบบต้องเก็บ **Log ประวัติการแก้ไข**
- **Asset Registry:** ฝ่ายทะเบียนสามารถเข้ามาดึงไฟล์เอกสารที่ผ่านการตรวจรับแล้วไปขึ้นทะเบียนครุภัณฑ์ต่อได้

## 5. Dashboard & Reporting

### 5.1 User Dashboard (หน่วยงานผู้ขอ)

- แสดงจำนวนโครงการทั้งหมดในปีงบประมาณ
- แสดงจำนวนโครงการที่กำลังดำเนินการ (On-going)
- มูลค่ารวมตามวงเงินงบประมาณ

### 5.2 Admin/Head Dashboard (ภายใน)

- **KPI Tracking:**
    - Cycle Time รวม (ตั้งแต่เริ่มจนจบ PO)
    - แยก Duration: ช่วงจัดซื้อ vs ช่วงตรวจรับ
    - จัดอันดับเจ้าหน้าที่ (Ranking) ตามประสิทธิภาพ
- **Alerts:** แจ้งเตือนสัญญาที่ใกล้ครบกำหนด (Recheck Due Date)

## 6. Non-Functional Requirements & UX/UI

- **UX/UI:**
    - ตัวอักษรต้อง **สีเข้ม/หนา** (Readable) ตัวใหญ่ อ่านง่าย
    - หน้าจอ Timeline ต้องดูง่าย แบ่งสีสถานะชัดเจน
- **Infrastructure:**
    - Server/Cloud: (รอสรุปกับ IT - ใช้ Server มหาลัยหรือเช่า Cloud)
    - Email: เชื่อมต่อ SMTP ของมหาลัย (รองรับ Volume ~3,000 โครงการ/ปี x 12 งวด)
- **History & Logs:**
    - ระบบต้องเก็บ Timestamp จริงของการทำงาน (Log System Time) แต่ User สามารถ "เลือกวันที่ (Manual Date)" ในแบบฟอร์มได้ (เพื่อลงวันที่ย้อนหลังตามเอกสารจริง)
